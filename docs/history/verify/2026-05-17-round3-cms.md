### Q1 — 환불 함수 권한 super_admin 체크 없음

- `admin-expense-refund.ts` 3·13행: `requireAdmin` + `guardFailed` 패턴만 사용, `isSuperAdmin` 코드 없음
- `admin-revenue-refund.ts` 3·13행: 동일 패턴

### Q2 — 자동 발송 트리거 cooldown 중복 차단

`cron-auto-trigger-evaluator.ts` 45-71행:
- cooldownDays = `trigger.cooldown_days ?? 30`
- cooldownCutoff = `Date.now() - cooldownDays * 86400_000`
- `communication_auto_trigger_runs` 에서 cooldown 내 `status='ok'` 이력 SELECT
- 이력 있으면 → `runStatus='skipped'` + `status='skipped'` INSERT 후 `continue` (발송 실행 건너뜀)
- 이력 없으면 → 정상 발송 실행

### Q3 — donations.paid_at 기존 row 백필

- `033ee0d chore(round3-cms): 마이그 파일 삭제 (호출 완료 2026-05-17)` 커밋 확인
- 마이그레이션 SQL: `UPDATE donations SET paid_at = COALESCE(hyosung_paid_date, created_at) WHERE paid_at IS NULL;`
- 호출 완료 후 파일 삭제 — 백필 실행됨

### Q4 — 신규 후원 생성 시 paid_at 자동 채움

donations 테이블 status enum: `pending | completed | failed | cancelled` (설계서의 'paid'는 'completed'와 동의어)

- **토스 결제 완료** (`donate-toss-confirm.ts` 118행): `paidAt: donation.updatedAt` ✅
- **토스 환불 후 재처리** (`donate-toss-confirm.ts` 295행): `paidAt: updated.updatedAt` ✅
- **어드민 수동 확인** (`admin-donation-confirm.ts` 227행): `paidAt: row.paymentDate ? new Date(row.paymentDate) : new Date()` ✅
- **효성 가져오기** (`admin-donation-confirm.ts` 290행): `paidAt: p.parsedDate || new Date()` ✅

### Q5 — 재정 집계 paid_at 기준 적용

- `admin-donations.ts` 115-122행: `COALESCE(paid_at, CASE WHEN pg_provider ILIKE '%hyosung%' THEN COALESCE(hyosung_paid_date, created_at) ELSE created_at END)`
- `admin-finance-income-summary.ts` 31행: `const paidAt = sql\`COALESCE(\${donations.paidAt}, \${donations.hyosungPaidDate}, \${donations.createdAt})\``
- 두 파일 모두 기간 필터·집계에 paidAt 기준 COALESCE 적용
