## 2. 워크플로우 동작 (코드 기준)

- **일시 후원**: 후원폼 → 거래등록(서버에 대기 후원 선저장·금액 고정) → KICC 결제창 → 복귀 시 서버가 저장 금액으로 승인 요청하고 KICC 승인금액과 대조 → 일치해야 완료·영수증·감사메일·포인트. 금액을 화면에서 조작해도 서버 저장값만 신뢰.
- **정기 후원**: 등록창 → 복귀 시 빌키 발급 → 즉시 1회차 청구 → 빌키 저장(다음 결제일 설정). 첫 결제 실패 시 빌키를 비활성으로만 기록하고 후원은 실패 처리.
- **자동청구(cron)**: 매일 새벽, 오늘이 약정일인 회원을 청구. 같은 회원이 정기·재시도 양쪽에 걸려도 1회만(중복청구 방지), 주문번호도 회원·월·차수로 고정해 PG단에서도 중복 차단. 실패는 1일·3일 뒤 재시도, 3회 실패면 자동 해지.
- **취소/환불**: 어드민이 환불 시 KICC 카드결제면 PG 실제 환불 호출, 효성·계좌이체는 상태만 변경(실제 환불 별도). 빌키 해지 시 KICC 빌키 삭제 호출(실패해도 내부 해지는 유지).
- **웹훅**: KICC 비동기 통지를 받아 후원 상태를 보조 동기화. 서명이 없는 규격이라 항상 정상 응답으로 재전송만 멈추고, 같은 상태면 무시(중복 안전).

## 3. 발견·조치 (verify/r40-kicc 직접 fix)

### [회귀·중간] 어드민 후원 환불 — KICC 자동환불 버튼이 안 뜸 → 복구
- **증상**: 어드민 후원 관리에서 KICC 카드결제 건을 환불하려 해도 "PG에서 실제 환불" 체크박스가 절대 노출되지 않아, 모든 환불이 "시스템 상태만 변경(수기)"로만 처리됨. 백엔드는 KICC 자동환불을 지원하는데 화면이 그 경로를 막던 형국.
- **원인**: 환불 모달이 옛 토스 기준 — `결제사 === 'toss'` 이고 토스 결제키가 있을 때만 자동환불 노출. PG가 'kicc'로 바뀌고 토스 결제키 항목이 사라지면서 조건이 항상 거짓이 됨 (admin-donations.js).
- **조치**: 노출 조건을 `결제사 === 'kicc'` 기준으로 변경(실제 거래번호 유효성은 백엔드가 후원ID로 재검증). 안내문·체크박스·확인창·버튼 라벨의 "토스" → "KICC" 정정. 죽은 토스 결제키 전달 경로(목록 버튼 data 속성 → 모달 인자 → hidden 입력) 제거. 환불 제출 키를 `autoRefundPg`(백엔드 신규 키)로 정정.
- **검증 한계**: 자동환불 버튼이 KICC 건에서 노출되고 올바른 요청을 보내는 것까지는 코드로 확인. **실제 KICC 환불 성사 여부는 Swain 브라우저+테스트카드 결제 후 환불 시나리오로 확인 필요.**

### [라벨] cms 카드 빌링 관리 탭 표시 "토스" → "KICC"
- 빌링키 관리 탭은 KICC 빌키 목록을 정상 조회·해지·재활성화·즉시청구함(`/api/admin/billing-keys` 호출). **기능은 멀쩡**하고 메뉴·제목 라벨만 "토스 자동결제/토스 빌링"으로 남아 사용자 혼란 소지 → 표시 텍스트만 "KICC"로 정정.
- 내부 식별자(탭 키 `toss-billing`·섹션 id `page-toss-billing`·함수명 `loadTbKeys`/`tb*`)는 화면-스크립트 연결 키라 **개명 시 동기화 회귀 위험**이 있어 그대로 둠 → 별도 정리 트랙은 메인 판단 권장.

## 4. 잔존(무해)·후속 권고

- **무해 잔존**(기능 영향 0): 라이브러리·cron 헤더 주석의 "토스 대체" 설명, 백엔드 환불 핸들러의 레거시 키 수용(`autoRefundToss`도 받아줌 — 의도적 하위호환), cms 내부 식별자, 메뉴얼 문서.
- **매뉴얼/AI 학습자료**: `public/manual-admin.html`의 `TOSS_MODE` 등 '토스' 표기 → 설계 §7 매뉴얼 갱신 트랙(별도). C 영역(코드) 밖이라 미수정·보고만.
- **실거래로만 확정되는 KICC 명세 미세조정**(lib/kicc.ts 주석에 명시): 인증서명(msgAuthValue) 평문 필드 순서, resCd 코드표, 에러 분류의 한글 키워드 매칭, returnUrl POST 필드 표기 — KICC 테스트키 실거래에서 응답을 받아봐야 확정 가능. **Swain 테스트 결제 시 함께 확인 권장.**
- **참고(설계 범위 외)**: 자동청구 대상이 약정일 "정확히 일치"만 잡아, cron이 하루 거르면 그 날짜 회원이 누락될 수 있음(`= today`). 기존 토스 cron과 동일 패턴이라 회귀는 아니나, 운영 안정성 차원에서 `<= today`(미처리분 포함) 전환을 차기 고려.

## 5. Swain 브라우저 확인 필요 (C 불가 — 카드·브라우저 없음)

KICC 테스트카드로 다음 흐름을 실제 진행해야 최종 확인:
- Q1 일시 후원 결제(결제창 이동→복귀→완료·영수증)
- Q2 정기 후원 빌키 등록(등록창→발급→첫 결제)
- Q3 정기 자동청구 cron 수동 트리거(약정일 청구·다음청구일 갱신)
- Q4 자동청구 실패→재시도(1일·3일·3회 후 해지)
- Q5 결제 취소/환불(어드민 환불 → KICC 실제 환불 성사)
- Q6 빌키 해지(KICC 빌키 삭제)
- Q7 웹훅 수신·중복(상태 동기화·중복 무시)

> 코드 경로는 모두 검증 완료. 위는 KICC 실거래 응답에 의존하는 부분으로, 코드만으로는 성사 여부를 단정할 수 없음.
