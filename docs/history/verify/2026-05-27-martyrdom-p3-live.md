> 검증: 2026-05-27 / C 채팅(Opus 4.7) / 브랜치 `verify/martyrdom-p3` (코드 변경 0 — 배포본 동작 확인만)
> 대상: 라이브 https://tbfa.co.kr (배포된 P3 · 실 DB · 실 AI(Gemini) · 실 R2)
> 목적: 코드 검증(PASS·무수정)을 넘어, 인증→API→DB→AI→파일 생성이 실제로 작동하는지 end-to-end 확인.
> 방식: 관리자 로그인(httpOnly 쿠키) 후 실 API 호출. 쿠키·자격증명은 저장소 밖 임시파일에 두고 검증 직후 삭제(본 보고서·커밋·메모리에 미기재).

## 1. 요약 — L1~L8 전부 PASS (실 데이터·실 AI 확인)

배포된 P3가 **실서버에서 인증·서면 생성·저장·내보내기·전문가 검토·패키지·권한 차단까지 완전히 작동**함을 확인했다. 응답은 mock이 아니라 **실 운영 DB·실 AI 출력**이었다(아래 근거).

| L | 시나리오 | 결과 | 실 데이터·mock 아님 근거 |
|---|---|---|---|
| L1 | 사건 목록 | ✅ PASS | 실 사건 2건(실명 사건)·자료 71/91건·hasExtraction 등 운영 데이터 |
| L2 | 목차 생성(실 AI) | ✅ PASS | AI가 **8개 섹션** 제안(기본 폴백 6개와 다름·사건 맞춤 sectionKey) |
| L3 | 섹션 본문 생성(실 AI) | ✅ PASS | 실 본문 생성(148자)·**자료에 없는 사실은 "(확인 필요)"로 표기**(환각 방지 가드 실동작) |
| L4 | 서면 로드 | ✅ PASS | 저장된 목차 8 + 섹션 8행(intro done·나머지 pending)·DB 왕복 일치 |
| L5 | 내보내기 PDF·docx | ✅ PASS | PDF 매직 `%PDF`(3.1MB·한글폰트 임베드)·docx 매직 `PK`(유효 zip)·한글 파일명 |
| L6 | 전문가 검토 배정·승인 | ✅ PASS | 배정(pending)→승인(approved)→**서면 상태 reviewed 전이**·decidedAt 기록 |
| L7 | 사건 패키지 zip | ✅ PASS | 매직 `PK`(3.4MB·자료 71건+분석+보고서 동봉)·한글 파일명 |
| L8 | 401 게이트 | ✅ PASS | 쿠키 없이 호출 시 **HTTP 401**(`관리자 로그인이 필요합니다`) |

## 2. 단계별 상세

### L1 — 사건 목록 (실 데이터)
- `GET /api/admin-martyrdom-cases` → HTTP 200·`ok:true`·total 2.
- 진행 사건 1건(active·자료 71건·구조 추출 완료) + 과거 학습사례 1건(reference·종결·자료 91건). 둘 다 P1/P2에서 운영자가 올린 **실 사건**(실명 포함 — 본 보고서엔 미기재).
- 검증 대상 = **진행 사건(자료 71건·구조 추출 있음)** 1건 선택.

### L2 — 목차 생성 (실 AI·1회)
- `POST /api/admin-martyrdom-draft-outline {caseId}` → HTTP 200·`ok:true`·`outputType:"draft"`·`status:"draft"`·서면 산출물 1건 생성.
- AI가 **8개 섹션** 제안: 신청 취지 → 고인·직무 개요 → 사망 경위·공무 관련성 → 직무상 과로·스트레스 → 의학적 인과관계·심리 요인 → 인정 요건 충족 → **예상 반론·반박** → 결론.
- 기본 폴백 목차(6개·고정 키)와 **다름** → 실제 AI가 사건 구조를 읽고 맞춤 생성한 근거(예상 반론 섹션 포함).

### L3 — 섹션 본문 생성 (실 AI·1회)
- `POST /api/admin-martyrdom-draft-generate {caseId, sectionKey:"intro"}` → HTTP 200·해당 섹션 `status:"done"`·본문 148자.
- **안전 가드 실동작 확인**: 본문이 고인 성명·학교명을 단정하지 않고 **"(확인 필요)"로 표기**. 설계 원칙("자료에 없는 사실 창작 금지")이 실 AI에서 그대로 작동.
- 근거(ragSources) 0건 — 신청 취지(intro)는 일반 법령 목적 진술이라 검색 매칭이 없던 것으로 보임(설계상 근거 없으면 일반지식 보강·단정 금지). 근거가 많이 필요한 섹션(의학·요건)은 비용 절감 위해 라이브 미생성.

### L4 — 서면 로드 (DB 왕복)
- `GET /api/admin-martyrdom-draft?caseId` → 저장된 목차 8섹션 + 섹션 8행(intro `done`·wordCount 148 / 나머지 7 `pending`) + 검토 0건.
- 생성·저장이 운영 DB에 정확히 반영됨(쓰기→읽기 일치).

### L5 — 내보내기 (실 파일 생성)
- PDF: `ok:true`·파일명 `유족급여신청서_MTR-…-0002.pdf`·`application/pdf`·base64 디코드 매직 `25504446`(`%PDF`)·**3,109,984바이트**(한글 폰트 NotoSansKR 임베드로 큼).
- docx: 파일명 `…-0002.docx`·`application/vnd.openxmlformats-…wordprocessingml.document`·매직 `504b0304`(`PK` = zip 컨테이너·유효 docx)·9,391바이트.
- 두 형식 모두 **한글 파일명**·유효 매직 헤더. (실제 한글 본문 렌더는 브라우저 열람으로 최종 확인 권장.)

### L6 — 전문가 검토 (상태 전이)
- 배정 가능 운영자 4명 조회 → 1명에게 배정: `POST review` → `ok:true`·검토 `pending`.
- 승인: `PATCH review {status:"approved", note}` → `ok:true`·검토 `approved`.
- **서면 산출물 상태가 `draft` → `reviewed`로 전이**·검토 행 `decidedAt` 기록·검토자 기록(승인 처리). 검토 워크플로 end-to-end 정상.

### L7 — 사건 패키지 zip
- `POST /api/admin-martyrdom-package {caseId}` → `ok:true`·파일명 `사건패키지_MTR-…-0002.zip`·`application/zip`·매직 `PK`·**3,366,385바이트**(자료 71건 원문 + 분석 산출물 + 보고서 PDF/HTML + README 동봉).
- 동기 생성(queued 아님)이며 자료 71건·보고서 PDF 포함에도 정상 응답.

### L8 — 권한 차단
- 쿠키 없이 `GET cases`·`GET draft`·`POST draft-outline` 호출 → 전부 **HTTP 401**·`{"ok":false,"error":"관리자 로그인이 필요합니다"}`.
- 운영자 인증 게이트가 실서버에서 정상 차단.

## 3. 생성된 흔적 (운영 DB·무해·운영자 폐기 가능)

라이브 검증으로 진행 사건(active·MTR-…-0002)에 다음이 생성됨:
- **서면 초안 산출물 1건**(목차 8섹션 + 본문 1섹션 생성·7섹션 pending). 검토 승인으로 상태 `reviewed`.
- **검토 행 1건**(승인 완료).
- 모두 운영자가 화면에서 폐기·재생성 가능한 무해한 초안. 새 사건은 만들지 않았고 기존 실 사건 위에서만 검증.

> ※ AI 호출은 목차 1회 + 섹션 1회로 제한(비용 통제). 내보내기·검토·패키지는 추가 AI 없음.

## 4. 결론

**배포된 P3는 실서버에서 인증→서면 생성(실 AI)→DB 저장→PDF/Word/zip 내보내기→전문가 검토 상태 전이→권한 차단까지 end-to-end로 정상 작동한다.** 코드 검증(무수정 PASS)에 이어 라이브 동작까지 확인 완료.

- 실 AI가 사건 맞춤 목차·본문을 생성하고, **자료에 없는 사실은 "(확인 필요)"로 표기**하는 안전 가드가 실동작.
- 한글 PDF·Word·zip이 유효한 파일로 생성(매직 헤더 확인).
- 전문가 검토 승인이 서면 상태를 `reviewed`로 전이.
- 인증 게이트가 비로그인 요청을 401로 차단.

남은 것은 운영자(Swain)의 브라우저 시각 확인뿐: PDF/Word를 실제로 열어 **한글 본문 레이아웃**과 섹션 순서, 논리맵 화면, 진행 오버레이 UX 정도. 기능적 동작은 라이브로 입증됨.
