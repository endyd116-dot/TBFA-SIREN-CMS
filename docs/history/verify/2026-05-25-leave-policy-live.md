**A-5 상세 (코드 정독)**
- **근속 산식**: `days = base + floor((근속년수-1)/incYears)*incDays`, 이후 상한(cap)으로 자름.
  - base12·inc1·incYears2 검산: 1주년 `12+floor(0/2)=12` / 3년차 `12+floor(2/2)=13` / 5년차 `12+floor(4/2)=14` → 명세 일치.
- **기존 잔여 보존**: 모드 B 부여는 `ON CONFLICT ... DO UPDATE SET total_days = GREATEST(기존, 신규)` — 적립분 손실 방지.
- **cron 모드 분기**: 만근 처리 루프는 모드 A일 때만, 근속 부여 루프는 모드 B일 때만 직원 순회(다른 모드면 빈 배열). "모드A=만근만 / 모드B=근속만" 일치.
- **임시 가드 제거**: `SERVICE_BASED_ANNUAL_ENABLED` 검색 결과 코드(.ts) 0건, 문서 3곳(HANDOFF·PROJECT_STATE·설계서)에만 흔적 → 코드에서 완전 제거 확인.

👁 화면 라디오 토글 시 모드 A/B 파라미터 영역 표시 전환은 시각 확인 필요(코드상 `toggleLeaveModeUI` 배선 정상).

## B. 카드 만료 유효기간 입력 (결제 도메인)

| 항목 | 결과 | 방식 | 비고 |
|---|---|---|---|
| B-1 저장 경로·회귀 격리 | ✅ PASS | 📖 | 후원ID → 후원의 빌키ID 조회(스키마 기반) → 빌키 테이블은 **raw SQL UPDATE만**(스키마 정의 비의존)으로 만료월 저장. 만료월은 YYMM 4자리 형식. 빌키 테이블이 스키마 조회 경로에 없어 회귀 격리 확인 |
| B-2 화면 입력 박스·핸들러 | ✅ PASS | 📖 | billing-success 페이지에 MM/YY 입력 2칸·구분자·등록 버튼·메시지 영역 존재. 핸들러는 프론트 검증(월 1~12·연 2자리) 후 후원ID 있을 때만 전송. 프론트/백 검증 일치 |
| B-3 라우팅·게이트 | ⚠️ 권고 | 🌐+📖 | 라이브: 빈 입력→400, 가짜 후원ID→404("결제 정보를 찾을 수 없습니다"), GET→405. **정상 동작**하나 **인증 게이트 없음**(아래 권고) |

**B-3 권고 — 인증 게이트 부재 (저위험·설계상 의도)**
- 트리거는 "401 게이트"를 기대했으나, 실제 이 API는 **로그인 검사 없이** 동작한다(후원ID→활성 빌키 매칭만 확인, 없으면 404).
- 코드 주석에 **의도와 근거가 명시**되어 있음: "만료월은 비밀정보가 아니고(카드번호·CVC 아님) 본인이 막 결제한 화면에서의 입력이라 매칭만 확인. 잘못 입력해도 만료 안내 알림만 빗나갈 뿐."
- **위험도 저평가 근거**: 카드번호·CVC 등 민감정보 노출 없음, 실제 청구·결제에 영향 없음, 최악의 경우 만료 사전안내 알림이 어긋나는 수준.
- **잠재 노출**: 후원ID가 순번 정수라 추정 가능 → 제3자가 임의 후원의 만료월을 오설정해 알림을 어긋나게 만들 여지(데이터·금전 피해는 아님).
- **권고**: 현 상태로도 동작·안전상 큰 문제는 아님. 다만 운영자 자립·악용 방지 차원에서 ① 후원 직후 일회성 토큰 검증 ② 또는 호출 빈도 제한 중 하나를 차후 보강할지 Swain 판단 권고. **인증/보안 변경이라 C가 임의 수정하지 않음.**
- 실제 빌키 저장 흐름은 R40 결제 실거래 시 Swain 검증 몫(빌키가 있어야 저장 가능).
