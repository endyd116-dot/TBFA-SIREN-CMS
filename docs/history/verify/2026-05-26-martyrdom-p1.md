## 2. 발견·수정한 결함 6건 (커밋 `fe6fe59`)

### BUG-1 [치명] 자료 처리 자동 체인이 한 번도 시작되지 않음
- **증상**: 운영자가 자료를 업로드해도 글자 추출·분류·색인·사건구조 추출이 전혀 일어나지 않고, 자료가 "처리 중"에서 멈춤(추출 함수 로그 비어 있음).
- **원인**: 업로드 완료 통지·수동 텍스트 입력·재분석 요청이 **뒤편 처리 함수를 호출만 하고 응답을 기다리지 않은 채(await 없이) 즉시 종료**. Netlify 함수는 응답을 반환하면 실행 환경이 멈춰, 보내지지 않은 호출이 전송 전 취소된다. 이는 **RAG 라운드에서 똑같이 겪고 고친 패턴(커밋 5313ce8)**이 4곳에서 재발한 것.
  - 해당 4곳: 업로드 완료 통지(`admin-martyrdom-doc-register`) → 추출, 추출 완료 후 → 사건 구조 분석(`admin-martyrdom-extract-background`), 수동 텍스트 입력(`admin-martyrdom-doc-reclassify`) → 재추출, 수동 재분석(`admin-martyrdom-reanalyze`) → 분석.
- **영향**: P1-2·P1-3·P1-4·P1-5·P1-9·P1-11(모든 추출·분류·구조추출 시나리오) + ★점검② 자동 이어짐.
- **수정**: 4곳 모두 뒤편 함수 호출을 **기다리도록(await)** 변경. 뒤편 함수는 즉시 접수 응답(202)을 돌려주고 실제 처리는 15분 한도로 계속되므로 화면 응답은 느려지지 않음. 호출 성공 여부를 응답(`bgStatus`)으로 노출해 네트워크 탭에서 확인 가능.

### BUG-2 [치명] 과거 학습사례 자료가 교차 검색용 코퍼스에 안 들어감
- **증상**: 과거 인정/불인정 사례(reference)를 올려도 다른 사건이 학습 근거로 찾을 수 없음.
- **원인**: 색인 코퍼스 종류가 **사건 종류와 무관하게 "지원 중 사건(martyr_active)"으로 고정**되어 있었음. 설계(§1.2)는 reference 자료는 "과거 사례(martyr_case)"로 색인해 모든 사건이 교차 검색하도록 규정.
- **영향**: P1-6(과거 사례 인정 자료 → `martyr_case` 색인) 깨짐. 딥릴리프의 "인정받는 증거 패턴" 학습 자산(§8.1·§9.2)이 전혀 쌓이지 않음.
- **수정**: 사건 종류가 reference면 `martyr_case`, 아니면 `martyr_active`로 색인(재색인 삭제 조건도 동일하게 분기). 두 경우 모두 사건 식별자로 격리.

### BUG-3 [중] 자료 상태 표시가 화면과 어긋나 자동 갱신이 멈춤
- **증상**: 자료를 올린 뒤 상태가 자동으로 "완료"로 바뀌지 않고(폴링 미작동), 상태 배지에 영문(`queued`·`extracting`)이 그대로 노출.
- **원인**: 백엔드가 설계 enum(`pending|processing|done|failed`) 밖의 값(`queued`·`extracting`)을 사용. 화면은 설계대로 `pending`/`processing`만 "진행 중"으로 인식해 자동 새로고침을 시작하지 않음.
- **영향**: ★점검-나(처리 상태 가시화) 약화. 운영자가 수동 새로고침해야 결과 확인.
- **수정**: 백엔드 상태값을 설계 enum대로 `processing`으로 통일(접수·진행 모두). 진행 배너·자동 폴링·배지 한국어 라벨 정상화.

### BUG-4 [중] 완료/실패 알림이 항상 실패(조용히 삼켜짐)
- **증상**: 추출 완료·실패 시 담당 운영자·총괄에게 알림이 전혀 가지 않음.
- **원인**: 알림 저장이 **알림 테이블에 존재하지 않는 칸(member_id·notification_type·meta)** 을 사용해 저장이 항상 실패. 실패는 try/catch로 조용히 무시되어 겉으로 안 드러남.
- **영향**: ★점검-가(이벤트 알림·완료/실패) 미작동.
- **수정**: 실제 알림 테이블 칸(recipient_id·recipient_type='admin'·category·ref_table·ref_id)으로 정정. recipient_type='admin'은 기존 운영자 알림 관습(admin-milestone-*)과 일치.

### BUG-5 [경] 과거 사례 인정 패턴을 엉뚱한 기준으로 추출
- **증상**: 진행 중 사건(active)에도 "인정/불인정 패턴" 항목이 추출 프롬프트에 섞이고, 과거 사례는 반대로 누락될 수 있음.
- **원인**: "인정 패턴" 추출 여부를 **사건 종류가 아니라 '신청서 문서 보유' 휴리스틱**으로 판정. 진행 중 사건도 순직신청서를 갖고 있어 오판.
- **영향**: 설계(§2.5: reference에서만 recognitionPattern) 위반. 기능 중단은 아니나 데이터 의미 오염.
- **수정**: 사건 종류(`caseKind='reference'`)를 분석 함수에 전달해 정확히 판정(미전달 시 기존 휴리스틱 폴백).

### BUG-6 [경] 새 사건 생성 후 자동 선택 실패
- **증상**: 새 사건을 만들면 목록에는 뜨지만 그 사건이 자동으로 열리지 않음.
- **원인**: 화면이 생성 응답에서 `d.case.id`(mock 형태)를 봤으나, 서버는 설계(§2.3)대로 **최상위 `id`·`caseNo`** 를 반환.
- **영향**: ★점검① 새 사건 생성 — 사건 자체는 정상 생성·목록 표시되나 자동 진입만 누락(경미한 UX).
- **수정**: 화면이 `d.id`를 우선 읽도록(폴백 `d.case?.id` 유지). 캐시버스터 `admin-martyrdom.js` v2→v3.

## 6. ★점검 포인트 4종

1. **새 사건 생성 응답** — 서버는 최상위 `id`·`caseNo` 반환(§2.3 정합). 화면 자동 진입이 mock 형태(`d.case.id`)를 보던 문제는 **BUG-6 수정**으로 해소.
2. **업로드 → 글자추출 → 8대분류 → 색인 → 사건구조 자동 이어짐** — 수정 전에는 **첫 단계부터 끊겨 있었음(BUG-1)**. 4곳 await 수정으로 체인 연결. 실제 흐름은 Swain 라이브.
3. **자료 행: 분류 배지·확신도·상태/재시도·원문 뷰어** — 8대분류 드롭다운·확신도(70 미만 "확인 필요")·실패 행 재시도·텍스트 직접 입력·원문 뷰어(PDF/이미지 인라인) 정합. 상태 자동 갱신은 **BUG-3 수정**으로 정상화.
4. **통합 CMS 🕊️ 순직 인정 지원 메뉴 로드** — 사이드바 항목(`data-tab="martyrdom"`)·iframe 페이지(`id="page-martyrdom"`·`data-nf-src="/admin-martyrdom.html"`) 등록 확인. 자동 매핑으로 로드. ✅
