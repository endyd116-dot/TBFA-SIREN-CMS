## 핵심 진단

R1 검증에서 드러난 두 경계면 결함(① 화면이 가짜 데이터로 그려짐 `USE_MOCK=true` ② 응답 키 불일치)이 **R2에서는 재발하지 않았다.**

- **mock 토글 없음**: R1처럼 항상 가짜를 그리는 `USE_MOCK=true`가 없다. mock은 `res.ok ? 실데이터 : MOCK`·`catch → MOCK` 형태의 **장애 시 폴백**으로만 존재. 정상 동작 중에는 실제 백엔드 데이터를 그린다(라이브로 확인).
- **응답 키 1:1 정합**: 백엔드 9개 API의 응답 키가 설계서 §2 명세와 일치하고, 프론트의 `unwrap()`/`pick()` 2단계 언랩(`res.data.data[k] || res.data[k]`)이 모든 호출에서 그 키를 정확히 읽는다.
- **라이브 정상**: 공개 API 3종이 실제로 교사 8분·히어로·카운터를 반환. 백엔드·시드·화면이 같이 실행되어 정합.

남은 사안은 **추모관이라는 민감한 맥락에서의 폴백 품질** 한 건뿐(아래 권고).

## 개선 (P1 — 추모 맥락 품질 · Swain 승인·fix 적용 완료 2026-05-24)

### mock 폴백이 장애 시 가짜 고인·가짜 추모 수치를 노출

- **현상**: 정상 동작 중에는 실제 데이터만 표시(라이브 확인). 그러나 공개 API가 일시 장애(HTTP 비200 또는 네트워크 오류)면 화면이 **하드코딩 mock으로 폴백**:
  - `memorial.js` loadSummary 실패 → 카운터 `1280명 / 3540 / 870` (MOCK_SUMMARY)
  - `memorial.js` loadTeachers 실패 → 그리드에 **`故 ○○○ 선생님` "아이들을 사랑한 선생님"** (MOCK_TEACHERS)
  - `memorial-teacher.js` 네트워크 오류 → **`故 ○○○ 선생님`** 가짜 프로필 (MOCK_TEACHER)
  - `memorial-bgm.js` 요약 호출 실패 → 더미 트랙 1개로 토글 노출
- **왜 신중히 봐야 하나**: 추모관은 실명·실제 사건 기반 공간(유가족 동의 확보). 장애 시라도 **존재하지 않는 가짜 고인·가짜 추모 인원이 공개되는 것은 부적절**하며 유가족에게 상처가 될 수 있다. CLAUDE.md §6.18(모든 콘텐츠 어드민 관리·하드코딩 금지)와도 충돌.
- **R1 BUG와의 구분**: R1은 `USE_MOCK=true`로 **항상** 백엔드를 우회한 명백한 동작 BUG였다. R2는 정상 시 실데이터를 그리므로 **동작 BUG가 아니라 장애 시 폴백 품질 이슈**다. 그래서 자의적으로 고치지 않고 보고한다(설계서 §5.1에 mock이 명시되어 있어 A의 의도적 폴백 — 로직 변경은 §6.10상 확인 영역).
- **적용한 fix(Swain 승인)**: 장애 시 mock 대신 이미 구현된 **빈 상태(empty state)**·카운터 0·not-found로 폴백하도록 교체하고 MOCK_* 상수 전부 삭제.
  - `memorial.js`: MOCK_SUMMARY/TEACHERS/MESSAGES 삭제 → loadSummary 실패 시 카운터 0·히어로 미표시, loadTeachers 실패 시 빈 그리드(empty state), loadMessages 실패 시 빈 방명록(empty state)
  - `memorial-teacher.js`: MOCK_TEACHER/MESSAGES/LETTERS 삭제 → loadTeacher 실패 시 not-found, 메시지·편지 실패 시 빈 목록
  - `memorial-bgm.js`: MOCK_TRACKS 삭제 → 요약 조회 실패 시 음원 0개와 동일하게 토글 숨김
  - 캐시버스터: memorial.html·memorial-teacher.html의 memorial.js·memorial-teacher.js·memorial-bgm.js `?v=1`→`?v=2`
  - 정상 동작 경로(실데이터 렌더)는 불변 — 장애 폴백만 변경. node --check 3종 OK.
