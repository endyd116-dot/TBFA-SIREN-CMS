## §1. DB 설계

### 1.1 donations — paid_at 컬럼 추가 (마이그)

```sql
ALTER TABLE donations ADD COLUMN IF NOT EXISTS paid_at TIMESTAMP;

-- 기존 데이터 백필
UPDATE donations
SET paid_at = COALESCE(hyosung_paid_date, created_at)
WHERE paid_at IS NULL;
```

**paid_at 기준 정책**:
- 효성 CMS+: `hyosung_paid_date` (실제 입금일)
- 토스 자동청구: `created_at` (승인 시각)
- 계좌이체: `created_at` (등록 시각)
- 신규 후원 생성 시: `created_at` 자동 채움 (status='paid'인 경우만)

**schema.ts 변경**: donations 테이블에 `paidAt: timestamp("paid_at")` 추가 (append-only)

### 2.2 [수정] 자동 발송 중복 차단 쿼리

cron 또는 자동 트리거 발송 함수에서 실행 전 체크:
```typescript
// communicationAutoTriggerRuns에서 cooldown 내 실행 이력 확인
const recentRun = await db.select()
  .from(communicationAutoTriggerRuns)
  .where(
    and(
      eq(communicationAutoTriggerRuns.triggerId, trigger.id),
      gte(communicationAutoTriggerRuns.triggeredAt,
          new Date(Date.now() - trigger.cooldownDays * 86400_000)),
      eq(communicationAutoTriggerRuns.status, 'ok')
    )
  ).limit(1);

if (recentRun.length > 0) {
  // 이미 cooldown 내 실행됨 → 스킵 + status='skipped' 기록
  return;
}
```
적용 위치: `cron-communication-auto-trigger.ts` 또는 해당 cron 함수.

### 2.3 [수정] donations 조회 API — paidAt 기준 적용

donations INSERT 시 paid_at 자동 채움:
```typescript
// 후원 완료(status='paid') 시 paidAt 설정
paidAt: status === 'paid' ? (hyosungPaidDate || new Date()) : null
```

재정 리포트 쿼리에서 `createdAt` → `paidAt` 기준으로 교체 대상 파일:
- `admin-donate-list.ts` — 기간 필터
- `admin-income-stats.ts` — 통계 집계
- `cron-billing-monthly.ts` — 월별 집계

### 2.4 [신규] POST /api/admin-billing-key-reactivate

```
POST /api/admin-billing-key-reactivate
body: { billingKeyId: number, reason: string (5자+) }

동작:
1. requireAdmin
2. billingKeys 조회 — isActive=false 확인
3. UPDATE: isActive=true, deactivatedAt=null, deactivatedReason=null, updatedAt=now()
4. notifications INSERT (해당 회원에게 "정기후원이 재활성화되었습니다")
5. logAdminAction 감사 로그
6. 응답: { ok, data: { billingKeyId, memberId, message } }

export const config = { path: "/api/admin-billing-key-reactivate" }
```

**사유 검증**: reason 5자 미만 → 400

### 2.5 [수정] 채널·템플릿 호환성 검증

발송 API (`admin-notification-send.ts` 또는 유사 파일) 발송 전:
```typescript
const VALID_CHANNELS = ["email", "sms", "kakao", "inapp"] as const;
// channels 파라미터가 VALID_CHANNELS 외 값 포함 시 400
const invalid = channels.filter(c => !VALID_CHANNELS.includes(c));
if (invalid.length) return badRequest(`지원하지 않는 채널: ${invalid.join(', ')}`);

// template.channel 이 있고 channels 배열에 없으면 경고 (발송은 계속)
// 응답에 warnings: string[] 추가
```
