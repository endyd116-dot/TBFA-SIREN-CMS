# 전자세금계산서 발급/매입 모듈 개발정의서

## 0. 문서 개요

| 항목 | 내용 |
|---|---|
| 문서명 | 전자세금계산서 발급/매입 모듈 개발정의서 |
| 대상 시스템 | 공유오피스 운영용 통합 CMS/MIS |
| 모듈 위치 | 회계/청구 모듈 하위 (기존 시스템과 연계) |
| 외부 연동 | ASP(공인전자문서중계자) API — Popbill 또는 바로빌 |
| 법적 근거 | 부가가치세법 제32조, 부가가치세법 시행령 제68조, 국세기본법 |
| 보관 의무 | 전자세금계산서 5년 |

## 1. 프로젝트 배경 및 목적

### 1.1 배경
- 공유오피스 운영자가 입주사(이용자)에게 매월 정기적으로 세금계산서를 발행해야 함.
- 운영자 본인이 매입하는 임대료·통신비·청소·소모품 등의 세금계산서를 자동 수집해야 함.
- 회계 SaaS(자비스, 더존 등)를 별도 구독하지 않고, 자체 CMS/MIS에서 발급·수집·집계까지 통합 처리한다.

### 1.2 목적
- 본인 시스템 내에서 전자세금계산서 **발급, 매입 수집, 부가세 신고용 데이터 집계**까지 완결.
- 외부 회계 SaaS 의존도 제거.
- 부가세 신고 시 홈택스에 입력할 데이터를 자체 시스템에서 추출 가능.

### 1.3 비범위 (Out of Scope)
- 부가세 신고서 자동 제출 (홈택스 직접 신고로 처리)
- 종합소득세/법인세 신고서 작성
- 재무제표 자동 생성

## 2. 시스템 컨텍스트 (전제)

본 모듈은 기존 CMS/MIS의 다음 모듈과 연계된다.

| 기존 모듈 | 연계 내용 |
|---|---|
| 입주사 관리 | 거래처(공급받는자) 정보 제공: 사업자번호, 상호, 대표자, 주소, 이메일 |
| 멤버십/계약 관리 | 정기 청구 대상 및 금액 정보 제공 |
| 좌석/회의실 예약 | 종량제 사용분 청구 금액 계산 |
| 결제/자동이체 | 결제 완료 시 발행 트리거 (선택) |
| 권한 관리 | 발행 권한 분리 |

전제 데이터베이스(또는 동등한 테이블)가 이미 존재한다고 가정한다. 없다면 본 모듈에서 거래처 테이블을 자체 생성한다.

## 3. 기능 요구사항

### 3.1 전자세금계산서 발급 모듈

#### 3.1.1 단건 발급
- 운영자가 화면에서 거래처·품목·금액 입력 후 발행 버튼 클릭.
- 시스템이 ASP API를 호출해 발행 → 승인번호 수신 → DB 저장.
- 발행 즉시 거래처에 이메일/카카오 알림톡 자동 발송 (옵션 선택 가능).

#### 3.1.2 정기 발급 (배치)
- 매월 지정일(예: 1일 또는 말일) 자동 실행.
- 정기 청구 대상 입주사 전체에 동일 양식으로 일괄 발행.
- 발행 결과를 운영자에게 요약 리포트로 알림(성공 N건/실패 M건).

#### 3.1.3 발행 양식
- **공급자**: 운영자 본인 사업자 정보 (시스템 설정값).
- **공급받는자**: 입주사 정보 (사업자번호 검증 후).
- **품목**: 사전 정의된 카테고리에서 선택 가능.
  - "비즈니스센터 이용료"
  - "사무지원 서비스료"
  - "회의실 사용료"
  - "부가서비스료(우편/통신/비품 등)"
  - "기타 (직접 입력)"
- **공급가액·세액·합계**: 자동 계산 (부가세 = 공급가액 × 10%, 일반과세 기준).
- **작성일자**: 공급시기 기준 (월별 청구분은 해당 월 말일 권장).
- **비고**: 자유 입력.

#### 3.1.4 수정/취소
- 발행 후 정정이 필요한 경우 **수정세금계산서** 발행.
- 사유 코드 선택 필수:
  - 01: 기재사항 착오 정정
  - 02: 공급가액 변동
  - 03: 환입
  - 04: 계약 해제
  - 05: 내국신용장 사후 개설
  - 06: 착오에 의한 이중발급 등
- 원본 발행과의 연결관계 DB에 기록.
- 발행 후 7일 이내 취소 가능 (사유 코드 06).

#### 3.1.5 발행 검증
- 발행 전 거래처 사업자등록상태 자동 조회 (국세청 사업자상태조회 API).
- 폐업·휴업 사업자에게는 발행 차단(경고 후 강제 발행 옵션).
- 사업자번호 체크섬 자동 검증.

---

### 3.2 매입세금계산서 수집 모듈

#### 3.2.1 자동 수집
- ASP API의 매입세금계산서 조회 기능 사용.
- 일 1회 또는 시간당 배치 실행 (설정 가능).
- 본인 사업자번호로 발행받은 매입 건을 자동으로 DB 적재.

#### 3.2.2 수동 등록
- ASP에서 수집되지 않는 종이/PDF 세금계산서는 운영자가 수동 입력.
- 등록 시 적격증빙 여부 체크박스 (세금계산서/계산서/신용카드/현금영수증 구분).

#### 3.2.3 자동 분류
- 공급자 사업자번호 기준 카테고리 자동 매핑.
- 카테고리 예시:
  - 임대료
  - 통신비
  - 전기·수도·가스
  - 청소·관리비
  - 소모품·비품
  - 인테리어·시설
  - 광고선전비
  - 기타
- 카테고리 매핑 규칙은 운영자가 룰 설정 화면에서 관리.

#### 3.2.4 매입세액 공제 여부
- 매입 건별로 공제 가능 여부 플래그.
- 접대비·비영업용 차량 등 비공제 항목은 별도 표시.

---

### 3.3 거래처(공급받는자) 관리

#### 3.3.1 거래처 등록
- 사업자등록번호, 상호, 대표자, 사업장 주소, 업태/종목, 담당자 이메일, 담당자 휴대폰.
- 사업자번호 입력 시 국세청 사업자상태조회 API 자동 호출 → 정보 자동 채움.

#### 3.3.2 정기 발행 설정
- 거래처별로 정기 발행 여부, 금액, 품목, 발행일 지정.
- 종량제(좌석수·회의실 시간 등 변동분)도 지원.

#### 3.3.3 사업자상태 주기 검증
- 월 1회 등록된 거래처 전체에 대해 사업자상태 일괄 조회.
- 폐업/휴업 변경 시 운영자에게 알림.

---

### 3.4 알림 모듈

#### 3.4.1 발행 알림
- 발행 즉시 거래처 담당자에게 이메일 + 카카오 알림톡 발송.
- ASP API 통합 기능 사용 (Popbill의 카카오 알림톡 모듈 활용 가능).
- 이메일 본문에는 발행 정보 요약 + PDF 다운로드 링크.

#### 3.4.2 운영자 알림
- 정기 발행 배치 완료 시 결과 요약(성공/실패).
- 발행 실패 시 즉시 알림 (Slack/이메일 등).
- 매입 수집 시 신규 매입 건수 일일 리포트.

### 3.5 통계/대시보드

#### 3.5.1 매출 대시보드

- 거래처별 매출 순위.

#### 3.5.2 매입 대시보드

#### 3.5.4 신고용 자료 출력
- 분기별 매출·매입장 Excel/CSV 출력.

## 6. 외부 API 연동

### 6.1 ASP API (1순위: Popbill / 2순위: 바로빌)

#### 6.1.1 Popbill 기준 연동 함수 (참고)

```python
# 의사코드 - 실제 SDK 호출은 Popbill 공식 문서 참조
from popbill import TaxinvoiceService

service = TaxinvoiceService(LinkID, SecretKey)
service.IsTest = False  # 운영환경

# 발행
def issue_tax_invoice(invoice_data):
    taxinvoice = Taxinvoice(
        writeDate=invoice_data['issue_date'].strftime('%Y%m%d'),
        chargeDirection="정과금",
        taxType="과세",
        purposeType="영수",
        # 공급자
        invoicerCorpNum=COMPANY_BUSINESS_NO,
        invoicerCorpName=COMPANY_NAME,
        invoicerCEOName=COMPANY_CEO,
        invoicerAddr=COMPANY_ADDRESS,
        # 공급받는자
        invoiceeType="사업자",
        invoiceeCorpNum=invoice_data['counterparty_business_no'],
        invoiceeCorpName=invoice_data['counterparty_name'],
        invoiceeCEOName=invoice_data['counterparty_ceo'],
        invoiceeEmail1=invoice_data['counterparty_email'],
        # 금액
        supplyCostTotal=str(invoice_data['supply_amount']),
        taxTotal=str(invoice_data['tax_amount']),
        totalAmount=str(invoice_data['total_amount']),
        # 품목
        detailList=[{
            'itemName': invoice_data['item_name'],
            'spec': '',
            'qty': '1',
            'unitCost': str(invoice_data['supply_amount']),
            'supplyCost': str(invoice_data['supply_amount']),
            'tax': str(invoice_data['tax_amount']),
        }]
    )
    
    result = service.registIssue(
        CorpNum=COMPANY_BUSINESS_NO,
        Taxinvoice=taxinvoice,
        Memo="정기 청구"
    )
    return result  # 승인번호 포함 응답
```

    return result.list

### 15.1 사업자번호 체크섬 검증 알고리즘

```python
def validate_business_no(no: str) -> bool:
    """사업자등록번호 10자리 체크섬 검증"""
    if not no.isdigit() or len(no) != 10:
        return False
    weights = [1, 3, 7, 1, 3, 7, 1, 3, 5]
    total = sum(int(no[i]) * weights[i] for i in range(9))
    total += (int(no[8]) * 5) // 10
    check = (10 - (total % 10)) % 10
    return check == int(no[9])
```

## 16. 구현 시 추가 요청사항 (개발자에게)

본 정의서를 기반으로 구현 시 다음을 준수해 주세요.

1. **모든 발행 액션은 트랜잭션 처리**. ASP API 호출 성공 → DB 저장 실패 시 ASP에서 발행은 이미 됨. 이런 경우를 위해 **승인번호 수신 직후 별도 워커가 DB 저장 보장**(at-least-once).
2. **ASP API 호출은 별도 모듈로 추상화**. 향후 Popbill에서 바로빌로 갈아탈 수 있도록 인터페이스 분리(`TaxInvoiceProvider` 추상 클래스).
3. **테스트 모드 / 운영 모드 명확히 분리**. 환경변수로 토글 가능하게.
4. **백그라운드 작업 큐 필수**. 발행은 비동기 처리, 사용자에게는 즉시 "발행 요청됨" 반환 후 결과는 별도 알림.
5. **모든 외부 API 응답은 raw JSON으로도 저장**. 추후 분석/디버깅 용도.
6. **거래처 정보는 발행 시점에 스냅샷으로 invoice 테이블에 저장**. 거래처 정보 변경 시 과거 발행 내역이 영향받지 않도록.
7. **타임존은 모두 KST(Asia/Seoul) 기준**. 작성일자·발행시각 모두 한국 시간 기준.
8. **금액 컬럼은 모두 DECIMAL(15, 2)** 사용. float/double 금지.
