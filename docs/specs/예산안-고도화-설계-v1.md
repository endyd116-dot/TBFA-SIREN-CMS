> 부수적으로 기존 예산안(budgetPlans) 디벨롭 방향·통합 CMS 배치·구현 로드맵 포함.

## 0. 확정 결정 (2026-07-01 Swain) — 구현 기준

미결 질문 5개가 아래대로 확정됨. 구현은 이 결정을 따른다.

1. **권한 3계층** — 오퍼레이터(직원·알바·서포터즈) = 기안 / 어드민(국장) = 1차 결재 / 슈퍼어드민(이사장) = 최종 결재.
2. **결재자 지정 = 직책(role) 기반** — 특정 회원이 아니라 직책으로 지정(국장 2명이면 둘 중 아무나 승인, 인사 이동에도 결재선 유지).
3. **금액 구간 3단계**:
   | 금액 | 결재 흐름 |
   |---|---|
   | 30만원 미만 | 국장 단독 승인 |
   | 30만 ~ 300만원 | 국장 → 이사장 |
   | 300만원 이상 | 국장 → 이사장 (필요 시 이사회 안건 표시) |
4. **지출결의서 = 정식 결의번호(제YYYY-NNNN호) 발행 + 결재 완료 시점 PDF를 R2에 박제 영구 보관** (감사·공시 대응). 볼 때마다 재생성이 아니라 발행본 원본 보존.
5. **위임(전결·대결) 설정 주체 = 이사장(슈퍼어드민) 전용** — 결재자 본인 셀프 위임은 불가(재정 통제 우선).

> 위 결정으로 §3~§5 설계의 열린 선택지는 모두 닫힘. 남은 실무 변수(금액 숫자 미세조정 등)는 통합 CMS 결재라인 설정 화면에서 운영자가 직접 조정.

## 1. 개요·목표

현행 예산·지출 시스템은 편성·집행·전표·통장분류·리포트까지 갖췄으나, **승인 절차가 super_admin 단일 승인**에 머물러 있다. 사단법인 내부통제와 시도교육청 제출 신뢰성을 위해 다음 3가지를 도입한다.

1. **결재라인** — 금액 구간별로 몇 명이, 어떤 순서로 결재하는지 규칙화. 순차 승인·반려·이력.
2. **전결·대결** — 상위 결재자 부재 시 하위자 위임(전결)·대리 결재(대결). 기간·범위 지정.
3. **지출결의서** — 사단법인 표준 서식(결의번호·지출일자·계정과목·적요·금액·증빙·기안/결재란)을 expenses·vouchers 데이터로 생성하고 PDF 출력.

**목표**: 운영자가 코드 없이 통합 CMS에서 결재라인을 설정하고, 지출 건이 자동으로 올바른 결재선을 타며, 완료 시 지출결의서를 PDF로 뽑아 증빙철에 보관·제출할 수 있게 한다.

## 2. 현행 분석

### 2.1 지금 되는 것

| 영역 | 테이블 | 현행 승인 |
|---|---|---|
| 예산 편성 | `budgetPlans` / `budgetLines` | status(draft→submitted→approved/rejected), `submittedBy`/`approvedBy`/`approvedAt`/`rejectionReason` — **1인 승인** |
| 지출 기록 | `expenses` | status(draft→approved/rejected), `recordedBy`/`approvedBy`/`approvedAt` — **1인 승인** |
| 전표 | `vouchers` | status(draft→submitted→approved/rejected), `createdBy`(uid)/`approvedBy`(uid) — **1인 승인** |
| 통장 자동분류 | `bankTransactions` / `counterparties` | AI 추론 + 관리자 확정 |
| 계정과목 | `accountCodes` | 마스터 CRUD |
| 리포트 | (집계) | 손익·재무 보고서 |

### 2.2 안 되는 것 (이번 설계 대상)

- **다단계 결재 없음**: 모든 승인이 `approvedBy` 단일 컬럼. 담당자→팀장→국장 같은 순차 결재 불가.
- **금액 구간 라우팅 없음**: 10만원 건과 1,000만원 건이 같은 절차.
- **결재 이력 추적 없음**: 누가 언제 승인/반려했는지 단계별 기록이 없음(마지막 1명만 남음).
- **위임(전결)·대리(대결) 없음**: 결재자 부재 시 대신 결재할 규칙·권한 없음.
- **지출결의서 서식·PDF 없음**: 전표는 있으나 사단법인 "지출결의서" 정식 서식과 결재란 출력이 없음.

> v3 명세 §6.3에 "금액 한도별 자동 분기(100만/1,000만)" 개념은 있으나 **미구현**. 이 문서가 그 구현 설계다.

## 3. 결재라인 설계

### 3.1 개념 모델

- **결재라인(template)**: "어떤 문서 종류 + 어떤 금액 구간이면 → 어떤 순서로 누가 결재" 규칙. 운영자가 설정.
- **결재요청(instance)**: 실제 지출/예산 건에 결재라인을 적용해 생성된 진행 중 결재. 각 단계별 상태를 가짐.

템플릿 2계층 + 인스턴스 2계층, 총 4개 테이블(제안).

### 3.2 신규 테이블 제안 (미적용)

> ⚠️ 아래는 **제안**이다. schema.ts append-only 원칙에 따라 `/* === 예산안 고도화 R? 결재라인 === */` 헤더로 파일 끝에 추가하고, 마이그레이션 함수 호출·적용 확인 후에만 정의를 활성화한다.

#### `approval_lines` — 결재라인 템플릿(헤더)

```
id              serial PK
docType         varchar(20)   -- 'expense'|'voucher'|'budget'  (결재 대상 문서 종류)
name            varchar(100)  -- '지출 100만원 미만 라인'
minAmount       bigint        -- 금액 구간 하한(원, 이상). null=무제한 하한
maxAmount       bigint        -- 금액 구간 상한(원, 미만). null=무제한 상한
isActive        boolean default true
sortOrder       integer default 0
createdBy       integer       -- members.id
createdAt/updatedAt timestamptz
-- UNIQUE(docType, minAmount, maxAmount) 로 구간 중복 방지(권장)
```

#### `approval_line_steps` — 결재라인 단계(디테일)

```
id              serial PK
lineId          integer FK approval_lines(id) cascade
stepOrder       integer       -- 1,2,3... 결재 순서
approverType    varchar(20)   -- 'role'|'member'  (역할 지정 or 특정인 지정)
approverRole    varchar(30)   -- approverType=role: 'team_lead'|'director'|'representative' 등
approverMemberId integer      -- approverType=member: members.id
stepLabel       varchar(50)   -- '팀장'|'국장'|'대표' 화면 표기
isFinal         boolean default false  -- 최종 결재 단계 표시(선택)
-- UNIQUE(lineId, stepOrder)
```

#### `approval_requests` — 결재요청(인스턴스 헤더)

```
id              serial PK
docType         varchar(20)   -- 'expense'|'voucher'|'budget'
docId           integer       -- 대상 레코드 id (expenses.id 등)
lineId          integer       -- 적용된 approval_lines.id (스냅샷용, null 허용)
amount          bigint        -- 라우팅 근거 금액(스냅샷)
status          varchar(20) default 'pending'
                -- 'pending'|'approved'|'rejected'|'canceled'
currentStep     integer default 1   -- 현재 대기 중인 단계 순서
requestedBy     integer       -- 기안자 members.id
requestedAt     timestamptz default now
completedAt     timestamptz
createdAt/updatedAt timestamptz
-- INDEX(docType, docId), INDEX(status), INDEX(currentStep)
```

#### `approval_request_steps` — 결재요청 단계별 이력(인스턴스 디테일)

```
id              serial PK
requestId       integer FK approval_requests(id) cascade
stepOrder       integer
stepLabel       varchar(50)   -- 스냅샷
approverMemberId integer      -- 이 단계 결재자로 확정된 members.id (위임 반영 후 실제자)
delegatedFrom   integer       -- 전결/대결로 위임받은 경우 원 결재자 members.id (null=본인)
status          varchar(20) default 'waiting'
                -- 'waiting'|'approved'|'rejected'|'skipped'
decidedAt       timestamptz
comment         text          -- 결재 의견·반려 사유
-- UNIQUE(requestId, stepOrder)
```

### 3.3 금액 구간 매트릭스 예시 (운영자 설정, 시드 아님)

| docType | minAmount | maxAmount | 단계 |
|---|---|---|---|
| expense | 0 | 1,000,000 | 담당자 전결 (1단계) |
| expense | 1,000,000 | 10,000,000 | 팀장 → 국장 (2단계) |
| expense | 10,000,000 | (무제한) | 국장 → 대표 (2단계) |

> v3 §6.3의 100만/1,000만 경계와 정합. 실제 구간·인원은 운영자가 CMS에서 편집(§7). 위 값은 화면 초기 안내용 예시일 뿐 하드코딩 금지.

### 3.4 순차 승인 상태머신

```
[문서 상신]
  └ 금액으로 approval_lines 매칭 (docType + minAmount≤amount<maxAmount)
  └ approval_requests 생성(status=pending, currentStep=1)
  └ steps 각 단계 스냅샷 생성(step1=waiting, 나머지 waiting)
        │
        ▼
[단계 N 결재 대기]  ← 결재자에게 알림
  ├─ 승인 → step N = approved
  │        ├ 마지막 단계면 → request.status=approved, 문서 status=approved, completedAt
  │        └ 아니면 → currentStep = N+1 (다음 결재자 알림)
  └─ 반려 → step N = rejected
           └ request.status=rejected, 문서 status=rejected, rejectionReason 기록
              (반송: 기안자에게 반환 — 재상신 시 새 request 생성)
```

- **반송 정책**: 반려 시 문서는 draft로 되돌리고 request는 rejected로 종결(불변 이력). 재상신하면 새 request row가 생겨 이력이 누적된다(감사 추적성).
- **동시성**: 결재 처리는 `currentStep`과 해당 step의 `status='waiting'`를 조건으로 한 조건부 UPDATE(낙관적 잠금)로 중복 결재 방지.
- **기존 컬럼 호환**: 최종 approved 시점에 기존 `expenses.approvedBy/approvedAt`·`vouchers.approvedBy/approvedAt`도 함께 채워 리포트·기존 화면 회귀를 막는다(이중기록).

## 4. 전결·대결 설계 (위임 권한)

### 4.1 개념

- **전결(위임)**: 상위 결재자가 자신의 결재 권한을 하위자에게 위임. 위임받은 사람이 그 단계를 최종 처리.
- **대결(대리)**: 상위자 부재 시 대리로 결재하되, 사후 상위자에게 보고(라벨상 "대결" 표기, 권한은 위임과 유사).

### 4.2 신규 테이블 제안 (미적용)

#### `approval_delegations` — 결재 위임 규칙

```
id              serial PK
delegatorId     integer   -- 위임하는 상위 결재자 members.id
delegateId      integer   -- 위임받는 하위자 members.id
kind            varchar(10)   -- 'jeongyeol'(전결) | 'daegyeol'(대결)
docType         varchar(20)   -- 'expense'|'voucher'|'budget'|'all'
maxAmount       bigint        -- 위임 한도(이 금액 이하만 위임, null=무제한)
startAt         date          -- 위임 기간 시작
endAt           date          -- 위임 기간 종료
isActive        boolean default true
reason          text          -- '출장'|'휴가' 등
createdBy       integer       -- 설정한 super_admin members.id
createdAt/updatedAt timestamptz
-- INDEX(delegatorId, isActive), INDEX(startAt, endAt)
```

### 4.3 위임 적용 규칙

결재요청 생성 시(또는 각 단계 진입 시) 다음 순서로 실제 결재자를 확정한다.

1. 단계의 지정 결재자(role/member)를 1차 후보로 잡는다.
2. 그 후보를 `delegatorId`로 하고 **오늘이 startAt~endAt 이내 + isActive + docType 일치 + amount≤maxAmount**인 활성 위임이 있으면 → `delegateId`를 실제 결재자로 치환. `approval_request_steps.delegatedFrom`에 원 결재자를 기록.
3. 위임 체인은 **1단계만**(위임받은 자가 또 위임하는 재귀는 금지 — 감사 혼란 방지). 필요 시 별도 규칙으로 확장.
4. `kind='daegyeol'(대결)`이면 화면 결재란에 "대결" 표기 + 사후 상위자 알림.

> 위임 규칙 설정은 **super_admin 전용**(§8). 위임은 "권한 이관"이므로 오·남용 방지를 위해 기간·한도·사유를 필수로 둔다.

## 5. 지출결의서 설계

### 5.1 서식 필드

사단법인 표준 지출결의서 1건 = 다음 항목.

| 구역 | 필드 | 소스 |
|---|---|---|
| 머리 | 결의번호, 지출일자, 회계연도 | 자동생성 / `expenses.occurredAt`·`vouchers.voucherDate` / `fiscalYear` |
| 본문(명세행) | 계정과목(코드·명), 적요(내역), 금액, 비고 | `vouchers.accountCode/accountName/description/amount` 또는 `expenses`+`accountCodes` |
| 합계 | 지출 총액(한글 금액 병기) | 명세행 합 |
| 증빙 | 증빙 종류·번호·첨부 유무 | `vouchers.evidenceType/evidenceNumber/evidenceUrl` / `expenses.receiptUrl` |
| 거래처 | 지급처, (선택)계좌 | `payeeName` / `counterparties` |
| 결재란 | 기안자 + 단계별 결재자·직위·결재일 | `approval_requests`·`approval_request_steps` |

### 5.2 데이터 연계

지출결의서는 **신규 저장 테이블을 만들지 않고**, 기존 `vouchers`(우선) 또는 `expenses`를 원천으로 **뷰/생성 로직**으로 조립하는 것을 1안으로 제안한다. 결의서 고유값(결의번호·발행이력)만 최소 테이블로 둔다.

#### `expense_resolutions` — 지출결의서 발행 레코드(선택·제안)

```
id              serial PK
resolutionNo    varchar(30) unique   -- 결의번호 체계(§5.4)
fiscalYear      integer
sourceType      varchar(20)   -- 'voucher'|'expense'
sourceId        integer       -- vouchers.id | expenses.id
approvalRequestId integer     -- 연결된 결재요청(결재란 렌더 근거)
issuedBy        integer       -- 발행자 members.id
issuedAt        timestamptz default now
pdfUrl          varchar(500)  -- 발행 PDF R2 URL(선택)
createdAt       timestamptz
-- INDEX(fiscalYear), INDEX(sourceType, sourceId)
```

> vouchers에 이미 `voucherNumber(YYYYMM-NNN)`이 있으므로, 결의서를 전표와 1:1로 본다면 이 테이블 없이 voucherNumber를 결의번호로 재사용하는 경량안(2안)도 가능. **1안 권장**(전표≠결의서 분리·재발행 이력 확보). — Swain 확정 필요(§9).

### 5.3 PDF 생성 방식

이 프로젝트는 pdf-lib(폰트 임베드 정밀 서식)와 window.print(간편) 병행. 지출결의서는 **결재란 격자·도장칸이 핵심**이므로:

- **1안(권장)**: `window.print` + 전용 print CSS — 결재란 표·격자·직위/성명/일자 칸을 HTML 표로 그리고 인쇄. 서식 수정이 쉬움. 한글 폰트 이슈 없음.
- **2안**: pdf-lib + NotoSansKR — 서버에서 고정 서식 PDF 발행·R2 저장(발행본 보존이 필요할 때). 영수증 PDF와 동일 스택 재사용.

권장: **화면·인쇄는 1안, 제출용 발행본 보존이 필요하면 2안 병행**. `expense_resolutions.pdfUrl`은 2안 채택 시에만 채움.

### 5.4 결의번호 체계

`제YYYY-NNNN호` (연도별 4자리 일련). 예: `제2027-0042호`. 회계연도별 시퀀스로 채번, 발행 시점 확정·불변. (전표번호 `YYYYMM-NNN`과 별개 — 결의서 고유 채번.)

## 6. 기존 예산안(budgetPlans) 디벨롭 항목

v3 명세와 정합하는 강화 방향(제안, 우선순위순):

1. **예산 대비 집행률 실시간**: `budgetLines.plannedAmount` vs 해당 카테고리 `expenses` 실집행 합을 집계해 라인별·전체 집행률(%) 노출. (신규 테이블 불필요 — 집계 쿼리 + 뷰)
2. **예산 초과 경보**: 지출 상신·결재 시 잔액 검증(v3 R-011). 집행률 80%/100% 도달 시 CMS 대시보드 경보·알림. **결재라인과 연동**(초과 건은 상위 결재선 강제).
3. **4축 태깅 강화**: v3의 회계구분(GEN/GRANT)·기능(EXEC/ADMIN/FUND)·재원 축을 `budgetLines`/`vouchers`에 점진 도입(별도 라운드). 결재라인의 `docType`을 축과 조합하면 보조금(GRANT) 건은 별도 결재선 지정 가능.
4. **추경(revision) 이력**: 현재 `budgetPlans`는 연도당 1개(unique). 추경을 revision 번호로 관리하는 확장은 v3 §5.2 `budget.revision`과 정합(후속 라운드).

> 이번 라운드는 **1·2번(집행률·경보 + 결재선 연동)**을 우선, 3·4번은 후속 제안으로 표기.

## 7. 통합 CMS 화면·메뉴 배치 안

프로젝트 원칙(§6.18): 모든 기능은 통합 CMS(`cms-tbfa.html`) iframe 안 운영자 관리. 현행 "📊 재정 관리" 그룹에 다음을 **추가**한다.

### 7.1 메뉴 추가 (재정 관리 그룹)

현행 하위메뉴: 손익 현황 / 후원 결제 내역 / 후원 외 매출 / 지출 관리 / 예산 관리 / 지출 자동 추출 / 재무 보고서. 여기에:

- `#finance-approval-lines` — **결재라인 설정** (⚙️, super_admin 전용 노출)
- `#finance-resolutions` — **지출결의서** (📄, 목록/작성/PDF)
- (결재 대기함) `#finance-approval-inbox` — **내 결재함** (✅, 로그인 결재자별 대기 목록)

### 7.2 iframe 4곳 등록 (code_standards #4)

각 신규 화면은 통합 CMS 안에서 열리도록 4곳 등록:

1. **사이드바** `<li><a data-tab="...">` 항목 추가 (위 §7.1)
2. **section** `<section class="cms-page" id="page-...">` 추가
3. **tabLabels** 매핑(탭 라벨 사전)에 한글 라벨 등록
4. **탭 분기** 로직에 해당 탭 로드 스크립트 연결

새 JS: `admin-finance-approval.js`(결재라인·결재함), `admin-finance-resolution.js`(지출결의서·PDF). 캐시버스터 `?v=` 부여.

### 7.3 화면 구성

- **결재라인 설정**: docType 탭(지출/전표/예산) → 금액 구간 카드 목록 → 구간별 단계(드래그 순서, SortableJS) 추가/편집. 위임(전결·대결) 규칙 서브탭.
- **내 결재함**: 대기 건 리스트(금액·기안자·문서 링크) → 승인/반려(의견 입력). 위임받은 건은 "대결/전결" 뱃지.
- **지출결의서**: 전표/지출에서 "결의서 발행" → 목록에서 열람·인쇄(PDF). 결재란은 결재요청 이력 렌더.

## 8. 권한 (role_permissions 연계)

`role_permissions`(category 신설 `finance` 또는 기존 재정 키 확장)에 feature 시드(제안):

| featureKey | 라벨 | 기본 admin | 기본 operator |
|---|---|---|---|
| `finance_approval_config` | 결재라인·위임 설정 | ❌(super_admin만) | ❌ |

- **결재라인·위임 설정**: `super_admin` 전용. (feature 게이트 + super_admin 항상 허용 패턴)

## 9. 마이그레이션·구현 로드맵

> 원칙: schema.ts **append-only**(본인 섹션 헤더 후 파일 끝 추가), DB 적용 전 정의 활성화 금지, 마이그레이션 함수 호출·확인 후 활성화. 1회용 마이그 호출 후 삭제.

**단계 A — 결재라인 코어** (회귀 위험 낮음, 신규 테이블만)
1. `approval_lines`·`approval_line_steps`·`approval_requests`·`approval_request_steps` 마이그 작성 → 호출 → schema 활성화.
2. 지출(expenses) 상신 시 결재요청 생성·순차 승인 API. **기존 단일승인은 유지하되** 결재선 미설정 구간은 종전 방식으로 fallback(무중단).
3. CMS 결재라인 설정·내 결재함 화면(iframe 4곳).

**단계 B — 전결·대결**
4. `approval_delegations` 마이그 → 활성화. 결재자 확정 로직에 위임 반영. 위임 설정 화면(super_admin).

**단계 C — 지출결의서**
5. (1안 채택 시) `expense_resolutions` 마이그 → 활성화. 결의서 조립·발행·인쇄(print CSS) + 선택 pdf-lib 발행본.

**단계 D — 예산 디벨롭**
6. 집행률·초과 경보 집계 + 결재선 연동(초과 시 상위선 강제).

**회귀 위험 관리**
- 최종 승인 시 기존 `approvedBy/approvedAt` 동시 기록(리포트·기존 화면 무회귀).
- vouchers `createdBy/approvedBy`는 uid(varchar), expenses/budgetPlans는 members.id(int) — **결재 테이블은 members.id(int) 통일** 권장, vouchers 연계 시 uid↔id 매핑 주의(§9 미결).
- drizzle 다중 leftJoin 금지 → 결재 이력은 separate query + Map 매칭.

## 10. 미결 질문 (Swain 확정 필요)

1. **결재 단계 인력 구성**: 실제 직위 체계가 "담당자 / 팀장 / 국장 / 대표(이사장)"가 맞는가? 각 금액 경계(10만/100만/1,000만 등)와 단계 수를 확정해야 초기 안내 매트릭스를 잡는다. (역할 지정 vs 특정인 지정 중 어느 방식 선호?)
2. **결재자 식별 기준**: 결재자를 **역할(role)**로 지정할지, **특정 회원(member)**으로 지정할지, 혼용할지. 사단법인 규모가 작으면 특정인 지정이 단순.
3. **지출결의서 저장 방식**: `expense_resolutions` 신규 테이블로 결의서를 전표와 분리(1안·재발행 이력) vs 전표번호를 결의번호로 재사용(2안·경량). 어느 쪽?
4. **PDF 발행본 보존 필요 여부**: 인쇄(print)만으로 충분한지, 아니면 발행 시점 PDF를 R2에 영구 보존(pdf-lib)해야 하는지. 보존 필요 시 스택·용량 고려.
5. **위임(전결/대결) 승인 주체**: 위임 규칙을 super_admin만 설정하는 게 맞는지, 아니면 상위 결재자 본인도 자기 부재 위임을 설정할 수 있어야 하는지(운영 편의 vs 통제).
