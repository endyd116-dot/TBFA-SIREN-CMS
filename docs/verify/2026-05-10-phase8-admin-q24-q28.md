## 2. Q24~Q28 라이브 검증

### 2-A. 임시 trigger·cleanup 패턴
- `POST /api/verify-trigger-q24-q28` (1회용) — auth 어드민 자기 자신을 target으로 4개 이벤트 dispatch, params에 `marker='TEST'` 포함
- `POST /api/verify-cleanup-q24-q28` (1회용) — `params_snapshot->>'marker' = 'TEST'` 행 + 인앱 notifications 정리
- 검증 종료 후 두 endpoint 모두 즉시 삭제 (1회용 보안 원칙)

### 2-B. 결과

| Q | 이벤트 | 정책 채널 | 결과 |
|---|---|---|---|
| Q24 | `billing.success` | inapp + email | ✅ 2건 sent (latency ~1.5s) |
| Q25 | `card.expiring` | inapp + email + kakao(placeholder) | ✅ 3건 sent (kakao는 어댑터에서 외부 발송 X, DB 기록만) |
| Q26 | `workspace.activity` | inapp | ✅ 1건 sent |

| Q28 | 강제 실패 케이스 | — | 🟡 정적 검증으로 갈음 (아래) |

**KPI 검증 (cleanup 직전 측정)**:
- inapp 채널: 3건 sent, 성공률 100%, 평균 1564ms
- email 채널: 3건 sent, 성공률 100%, 평균 1174ms
- kakao 채널: 1건 sent, 성공률 100%, 평균 0ms (placeholder는 외부 호출 없음)
- sms: 0건 (어떤 이벤트도 sms 정책 없음 — 정확)

**채널 정책 정확성**: 7건 분포가 [notify-events.ts:30-40](../../lib/notify-events.ts#L30-L40)의 `EVENT_CHANNEL_POLICY`와 정확히 일치 — 디스패처가 정책 라우팅을 정상 수행.

**비동기 타이밍**: dispatcher는 fire-and-forget이라 trigger 호출 직후엔 빠른 채널만 INSERT 완료. 3초 sleep 후엔 4건, 11초 후엔 7건 모두 sent. 어드민 화면에서 자주 새로고침으로 진행 상황 확인 가능.

### 2-C. Q28 강제 실패 정적 검증

[lib/notify-dispatcher.ts](../../lib/notify-dispatcher.ts) `_dispatch` 흐름:
- 어댑터 throw 시 `status=failed`, `attempt=N`, `next_retry_at = now + RETRY_DELAYS_MS[attempt]` (1s → 5s → 25s)
- attempt=3 도달 시 `status=dead` 전환
- [cron-notification-retry.ts](../../netlify/functions/cron-notification-retry.ts) — `WHERE status='failed' AND next_retry_at <= now()` 폴링하여 재시도
- dead 도달 시 어드민 메타-알림은 retry cron 또는 dispatcher 자체가 처리

→ 코드 흐름은 설계서 §3.4와 일치. A 영역이라 본 검증 직접 대상 아님 (실제 fail 트리거는 어댑터 강제 throw가 필요한데, 그건 추후 별도 부하 테스트로). 라이브 어드민 화면에 dead·failed 행이 들어왔을 때의 표시(빨간 status badge + 에러 메시지 컬럼 + 실패 사유 Top 5 차트) 동작은 화면 정적 분석으로 확인 — 정상.

### 2-D. cleanup 결과

- `notifications` 인앱 3건 삭제 (Q24·Q25·Q26)
