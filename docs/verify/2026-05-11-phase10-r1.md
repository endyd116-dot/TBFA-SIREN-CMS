## 3. 시나리오별 코드 정합성 (Q1~Q8)

세션 쿠키 흐름은 Swain 영역이라 코드 정합성으로 시나리오 통과 여부를 판정.

### Q1 — 목록 진입 → 시드 3건 표시 ✅ PASS

- 목록 화면 진입 시 자동으로 목록 조회 호출
- 백엔드는 비활성 제외 + 갱신일 내림차순 정렬 + 페이지네이션
- 시드 3건(월간 뉴스레터·일회성 공지·AI 트리거)이 모두 활성 상태(`is_active=true`)로 들어가 있음 → 첫 조회에서 3건 모두 표시

### Q2 — 신규 작성 후 다시 열기 동일 내용 ✅ PASS

- 신규 버튼 클릭 → 편집 페이지로 이동(쿼리 없음)
- 채널·카테고리·제목·본문·변수 정의 입력 후 저장 버튼 → POST 신규 생성 → 토스트 "템플릿이 등록되었습니다." → 0.6초 후 목록으로 돌아감
- 새 행이 목록 맨 위(updated_at DESC)에 추가됨
- 수정 버튼 클릭 → 편집 페이지가 단일 조회 응답을 받아 폼을 채움 (이름·카테고리·채널 라디오·제목·본문·변수 표 행들 모두 복원)

### Q3 — 미정의 변수 사용 시 저장 거부 ✅ PASS

이중 차단:
1. **클라이언트 사전 점검**: 본문·제목에서 `{{key}}`를 추출해 변수 정의 표의 키와 비교 → 미정의 발견 시 토스트 "본문에 정의되지 않은 변수가 있습니다: {{unknown}}" 표시 + 서버 호출 자체를 안 함
2. **서버 검증**: 클라이언트가 우회해서 호출해도 서버가 동일 검증으로 400 거부 (`step:"validate"`, 같은 메시지)

DB 저장 0.

### Q4 — 미리보기 → 변수 값 수정 → 결과 확인 ✅ PASS

- 미리보기 버튼 클릭 → 변수 정의 표를 기반으로 모달에 입력 칸 자동 생성. 각 칸은 sample 값으로 미리 채워짐
- 사용자가 값 수정 가능
- "치환 결과 보기" 버튼 클릭 → POST 미리보기 → DB 저장 없이 순수 치환 결과 + 미정의 변수 경고 응답
- 모달에 치환된 제목·본문 표시. 제목은 이메일·인앱 채널만 노출. 변수 값을 비워두고 sample도 없으면 빈 문자열 + warning 표시

### Q5 — SMS 채널 → 글자수 카운터 ✅ PASS

- SMS 라디오 선택 시 제목 입력 칸 숨김(이메일·인앱만 노출), 본문 하단 카운터 표시
- 본문 입력할 때마다 갱신: "현재 N자 / SMS 90자 / LMS 2000자". 2000자 초과 시 .over 클래스로 강조
- 저장 시 채널이 SMS면 제목은 NULL로 보내짐(서버는 SMS·카카오는 제목 NULL 허용)

### Q6 — 카카오 채널 → Aligo 안내 박스 ✅ PASS

- 카카오 라디오 선택 시 제목 칸 숨김, 안내 박스 표시
- 본문 정도만 본 시스템에서 관리하고, 실제 발송용 알림톡 템플릿 ID는 Phase 9-B Aligo 어댑터(외부 심사 통과 후 환경변수 등록)와 별개임을 운영자가 인지하도록 안내

### Q7 — 삭제 → soft delete → 비활성 표시 ✅ PASS

- 삭제 버튼 클릭 → confirm 확인창 → POST 삭제 → 행이 회색으로 갱신, "비활성" 배지 추가
- 백엔드는 `is_active=false`로 갱신만 (실제 행 삭제 X) + 변경자·변경시각 기록
- 비활성 포함 체크박스 켜면 다시 표시되고 회색 행으로 보임
- 비활성 포함 체크박스 끄면 자동으로 목록에서 빠짐

### Q8 — 채널·카테고리·검색·비활성 동시 적용 ✅ PASS

- 화면에서 4개 필터 변경 시 즉시 첫 페이지부터 다시 조회
- 백엔드는 모든 조건을 AND로 결합 (`is_active`, `channel`, `category`, `name ILIKE '%q%'`) + 페이지네이션
- 결과 0건이면 "조건에 맞는 템플릿이 없습니다." 안내 표시

## 5. 코드 정합성 추가 메모

설계서와 어긋나거나 주의할 만한 점:

- 변수 참조 검증이 클라이언트 + 서버 양쪽에 있어 우회 시도에도 안전
- 미리보기는 거부 없이 warnings로만 알림 (의도대로 — 작성 중 자유롭게 시도 가능)
- 응답 키 다중 fallback(`res.data?.data ?? res.data`)을 클라이언트 모든 곳에서 사용 — CLAUDE §6.2 준수
- 단계별 try/catch + step 라벨(`validate` / `insert` / `select_list` / `soft_delete` / `render` 등) + detail/stack 응답 — CLAUDE §6.2 준수
- 사이드바 변경은 1줄 추가만 — 기존 그룹 구조 영향 0
