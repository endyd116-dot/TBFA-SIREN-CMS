## 3. 시나리오별 코드 정합성 (Q1~Q9)

세션 쿠키 흐름은 Swain 영역이라 코드 정합성으로 시나리오 통과 여부를 판정.

### Q1 — 사이드바 → 수신자 그룹 → 시드 4건 표시 ✅ PASS

- 사이드바 운영 그룹 끝에 `[👥 수신자 그룹]` 메뉴 1줄 추가됨 (CLAUDE §6.6 SPA 외부 이동 패턴 준수)
- 목록 화면 진입 시 자동으로 명단 조회 호출
- 백엔드는 `is_active=true` 필터 + 갱신일 내림차순 정렬 + 페이지네이션
- 시드 4건(전체 활성 회원·정기 후원자·일시 90일·운영자)이 모두 활성 상태로 들어가 있어 첫 조회에서 4건 모두 표시
- 등급 시드는 마이그 호출 시 자동 보정으로 제외됐음(메인 보고 기록)

### Q2 — 신규 → 필터(상태=활성 + 유형=정기) → 미리보기 ✅ PASS

- 편집 페이지에서 필터 방식 선택 → 조건 2개 추가 → 미리보기 버튼
- 백엔드는 검증 통과 후 회원 SELECT를 동적 WHERE 조립으로 실행 → 인원 수 + 샘플 5명 + 한 줄 요약 반환
- 모달에 인원 수 + 샘플 5명 + 0명일 경우 경고 메시지 표시

### Q3 — 신규 → 수동 명단 → 회원 검색·추가 → 저장 ✅ PASS

- 편집 페이지에서 수동 명단 방식 선택 → 회원 검색 → 추가
- 저장 시 백엔드는 `criteria.type='manual'` + `memberIds` 검증
- **존재하지 않는 회원 ID 자동 검출** → 누락 ID 명단 함께 반환하여 클라이언트가 안내 가능
- 1000명 상한 보장
- 저장 후 목록에 새 행 추가됨, 인원 수는 추가된 회원 수와 일치(실재 회원만 카운트)

### Q4 — 기존 그룹 수정 → 조건 1개 삭제 → 저장 → 다시 열기 ✅ PASS

- 단일 조회 → 폼 채우기 → 조건 행 1개 삭제 → 저장
- 백엔드는 같은 검증 통과 + **자기 자신 제외 이름 중복 검사** + jsonb 컬럼 갱신
- 다시 열면 변경된 조건이 그대로 표시되고 인원 수도 갱신(동적 계산)

### Q5 — 신규 → 필터 0개로 저장 시도 → 거부 ✅ PASS

- 검증 헬퍼가 `criteria.filters.length === 0` 발견 → 400 거절 + 메시지 "최소 1개 이상의 조건을 추가해 주세요."
- DB 저장 0
- 클라이언트도 동일 검증으로 토스트 표시 가능

### Q6 — 화이트리스트 외 값 직접 API 시도 → 400 ✅ PASS

- 검증 헬퍼가 회원 상태 enum(`pending`/`active`/`suspended`/`withdrawn`) 외 값을 자동 차단
- 회원 유형·후원 상태도 동일 enum 차단
- 필터 field가 화이트리스트 9종 외이면 거절, op도 field별 허용 op 외이면 거절 ("허용되지 않은 field" / "허용되지 않은 op")
- 모든 거절 응답에 step=`validate` 라벨 + 구체 메시지

### Q7 — 회원 보기 모달 → 페이지네이션 ✅ PASS

- 회원 보기 버튼 → 그룹 ID로 회원 명단 호출
- 백엔드는 그룹 ID로 criteria 조회 → 동일 헬퍼로 실시간 회원 SELECT
- limit 기본 50명, 최대 200명 / offset 페이지네이션
- 응답에 총 인원 수와 현재 페이지 회원 명단 함께 반환 → 페이지네이션 동작 가능

### Q8 — 그룹 삭제 → soft delete → 비활성 포함 시 회색 ✅ PASS

- 삭제 버튼 → confirm → POST 삭제 → soft delete (`is_active=false`) + 변경자·변경시각 기록
- 비활성 포함 체크박스를 켜면 명단 조회에 `includeInactive=1`이 붙어 비활성 그룹도 반환
- 클라이언트가 비활성 행을 회색으로 표시

### Q9 — 동적 그룹 (회원 상태 변경 후 다시 미리보기) ✅ PASS

- **핵심 설계**: criteria만 저장하고 회원 ID 캐시는 안 만듦. 매 호출마다 회원 SELECT를 다시 실행
- 정기 후원자 그룹 미리보기 → N명
- 비활성 회원 1명을 활성으로 바꾼 뒤 다시 미리보기 → N+1명
- 시점이 다른 결과를 반영하는 진짜 동적 그룹 동작
- 같은 원리로 list 화면의 인원 수도 호출 시점에 동적 계산됨

## 5. 성능 점검 (설계서 §4.4)

세션 쿠키 없이 인증 거부까지의 baseline 응답 시간 측정. 실 DB SELECT는 인증된 세션이 필요해 Swain 클릭 검증 항목으로 분리.

| API | 1차 | 2차 | 3차 | baseline |
|---|---|---|---|---|
| 그룹 명단 조회 | 0.66s | 0.60s | 0.59s | ~0.6s |
| 미리보기 | 0.61s | 0.58s | 0.61s | ~0.6s |
| 그룹 회원 목록 | 0.68s | 0.58s | 0.59s | ~0.6s |
| 비교: 기존 R1 발송 템플릿 명단 | 0.64s | 0.61s | 0.58s | ~0.6s |

→ 함수 cold start + 어드민 가드까지 ~0.6초로 R1과 동등. 신규 함수의 등록·로딩 자체는 정상 범위.

**Swain 클릭 검증 권장(인증된 세션에서 실 DB SELECT 시간)**:
- 명단 조회는 그룹 5개 기준 0.5초 미만이어야 정상 (memberCount 동적 계산 5번 = 5번 SELECT)
- 미리보기는 1만 회원 규모에서 1초 미만이어야 정상
- 어느 한 항목이 3초 넘으면 R3 머지 전 인덱스·캐시 보강 검토 필요 (설계서 §4.4)

## 6. 코드 정합성 추가 메모

설계서와 어긋나는 점은 발견되지 않았으나, 다음은 운영 중 알아두면 좋은 사항:

- **활성 정기 후원자 정의**: 헬퍼는 `donations.type='regular' AND status='completed'`로 EXISTS 검사. 설계서 §2.4 description의 `status='active'`는 donations.status enum에 `active` 값이 없어 코드가 `completed`로 구현. 의미는 같음("결제된 정기 후원 보유")이지만 R3 발송 시점에 실제로 어떤 의미를 갖는지 운영자 확인 권장
- **블랙 처리 판정**: `(status='suspended' AND blacklist_reason IS NOT NULL)` — 일반 정지와 블랙 분리
- **필터 빌드**: criteria가 jsonb 컬럼에 그대로 저장되어 매 호출마다 동적 WHERE 조립 → R3 발송 시점에도 같은 헬퍼로 회원 SELECT 가능 (재사용성 양호)

## 7. Swain 직접 클릭 검증이 필요한 항목

세션 쿠키 흐름이 필요. 코드 정합성으로는 통과지만 다음은 클릭으로 최종 확인 권장:

1. 어드민 로그인 → 사이드바 [👥 수신자 그룹] 클릭 → 시드 4건 표시 + 인원 수
2. 신규 → 필터(상태=활성, 유형=정기) → 미리보기 모달의 인원·샘플 5명
3. 신규 → 수동 명단 → 회원 검색·추가 → 저장 → 목록에 새 행 + 인원 수 = 추가한 회원 수
4. 기존 그룹 수정 → 조건 1개 삭제 → 저장 → 다시 열기 시 변경 반영
5. 신규 → 필터 0개로 저장 시도 → 토스트 거부
6. 회원 보기 모달 → 페이지네이션 (50명 단위)
7. 그룹 삭제 → 회색 행 표시 (비활성 포함 체크 시)
8. 정기 후원자 그룹 미리보기 → 비활성 회원 1명 활성으로 바꾼 뒤 다시 미리보기 → 인원 1 증가 (동적 그룹 확인)
9. 명단·미리보기 응답 시간 0.5~1초 미만 (실 회원 규모에서)
