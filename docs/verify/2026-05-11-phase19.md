## 2. 발견한 BUG (4건)

### BUG-19-01 [HIGH] 스크립트 자체 — 로그인 본문 필드명 불일치

- **증상**: 어드민 로그인 호출 시 401 응답 "ID 또는 비밀번호가 일치하지 않습니다"가 반복. 자격증명이 맞아도 무조건 실패
- **원인**: 스크립트가 로그인 본문을 `{ email, password }`로 보냈으나, 서버 검증 규칙은 `{ id, password }`만 받아들임. `email` 필드가 무시되어 ID가 비어있는 상태로 검증 실패
- **수정**: 스크립트 본문 키를 `id`로 변경
- **파일**: `scripts/healthcheck.mjs`

### BUG-19-02 [HIGH] 스크립트 자체 — 7개 API 경로가 실제 등록 경로와 다름

- **증상**: 인증 전 13건 중 7건이 401이 아닌 **404**를 응답. 해당 경로가 존재하지 않음
- **원인**: 함수마다 두 가지 경로 규약이 섞여 있음 — 슬래시 스타일(`/api/admin/X`)과 하이픈 스타일(`/api/admin-X`). 스크립트는 7개를 잘못된 스타일로 등록
- **수정**: 함수 구현부의 실제 등록 경로 기준으로 매핑을 정리

| 스크립트 기존 경로 (404) | 실제 등록 경로 |
|---|---|
| /api/admin-me | /api/admin/me |
| /api/admin-members | /api/admin/members |
| /api/admin-donations | /api/admin/donations |
| /api/admin-donation-dashboard | /api/admin/donation-dashboard |
| /api/admin-incidents-list | /api/admin/incident-reports |
| /api/admin-harassment-list | /api/admin/harassment-reports |
| /api/admin-support-list | /api/admin/support |

- **파일**: `scripts/healthcheck.mjs`

### BUG-19-03 [HIGH] 라이브 회귀 — 후원 목록 조회가 500 에러

- **증상**: 어드민이 후원 관리 화면 진입 시 후원 목록 API가 500을 응답. 본문 에러:
  `TypeError [ERR_INVALID_ARG_TYPE]: The "string" argument must be of type string or an instance of Buffer or ArrayBuffer. Received an instance of Date`
- **원인**: 후원 목록 함수에서 통계용 SQL 템플릿에 시간 객체(오늘 자정·이번 달 1일)를 그대로 끼워 넣었는데, 드라이버 단계에서 문자열로 변환할 때 실패. (드라이버 버전 차이로 인한 회귀로 추정)
- **수정**: 시간 객체를 ISO 문자열로 먼저 변환 후 SQL에서 `::timestamp`로 캐스팅
- **파일**: `netlify/functions/admin-donations.ts`
- **영향**: 어드민 후원 관리 페이지(목록·통계 모두) 정상 복구

### BUG-19-04 [HIGH] 라이브 회귀 — 통합 대시보드 KPI가 500 에러

- **증상**: Phase 16 통합 분석 대시보드의 SIREN 신고 KPI 영역이 500을 응답. 본문 에러:
  `PostgresError: UNION could not convert type harassment_report_status to incident_report_status` (단계: select_siren)
- **원인**: SIREN 신고 3종(사건/괴롭힘/법률) 상태 컬럼을 UNION으로 합칠 때, 테이블별로 enum 타입이 다름. PostgreSQL은 다른 enum 사이에서 자동 변환을 하지 않아 쿼리 자체가 깨짐
- **수정**: 세 SELECT 각각에서 상태 컬럼을 `::text`로 캐스팅한 후 UNION 수행
- **파일**: `netlify/functions/admin-dashboard-kpi.ts`
- **영향**: 통합 분석 대시보드 KPI 정상 복구

- 캐시 적용 단독 효과를 명확히 분리하려면 워밍업 후 같은 API를 5~10회 연속 호출하는 별도 측정이 필요. 본 라운드에서는 **응답시간이 모두 3000ms 이내**임을 확인하는 수준에서 검증 완료
