## 3. 🚨 BUG 발견 + fix

### 3.1 `admin-send-job-create.ts` — 등록자 ID NULL 저장 (BUG-5 회귀 클래스)

**현상**: 발송 작업 등록 시 `communication_send_jobs.created_by`가 항상 NULL로 들어감. 누가 등록했는지 DB에 안 남음.

**원인**: 38줄에서 `(auth as any).admin?.id ?? (auth as any).user?.id ?? null` 사용. 둘 다 undefined인 경로(`requireAdmin`은 `{ ok, ctx: { admin: { uid }, member } }` 구조라 `auth.admin`이 아니라 `auth.ctx.admin.uid`).

```diff
-  const adminId = (auth as any).admin?.id ?? (auth as any).user?.id ?? null;
+  /* fix(R3): BUG-5 패턴 회귀 — auth.admin?.id / user?.id는 항상 undefined.
+     실제 어드민 ID는 auth.ctx.admin.uid에 있음. silent NULL 저장 방지. */
+  const adminId = (auth as any).ctx?.admin?.uid ?? null;
```

**클래스**: BUG-5(`admin-finance-*`)와 동일 패턴. CLAUDE.md §9.1.8 "헬퍼 함수 도입 직후 모든 사용처 1회 검증" 정책으로도 잡혔어야 했으나 R3 신규 작성 시 누락.

**런타임 영향**: INSERT는 성공(스키마 NULL 허용), 조용히 NULL 저장 → 감사 추적 무력화. 다른 동작에는 영향 없음.

**fix 커밋**: 본 verify 브랜치에 포함.

### 3.2 다른 함수 점검 결과

`admin-send-job-cancel`은 `cancelled_by` 컬럼 자체가 없어 어드민 ID 기록 안 함(현 schema와 일치 — 의도). 그 외 함수는 어드민 ID를 컬럼에 안 적어서 동일 패턴 부재.

## 4. 시나리오별 코드 정합성 (Q1~Q12)

세션 쿠키 흐름은 Swain 영역이라 코드 정합성으로 시나리오 통과 여부를 판정.

### Q1 — 사이드바 → 발송 작업 → 빈 목록 또는 기존 작업 ✅ PASS

- 사이드바 운영 그룹 끝에 `[📤 발송 작업]` 1줄 추가됨 (CLAUDE §6.6 SPA 외부 이동 패턴)
- 명단 화면 진입 시 자동으로 작업 명단 호출

### Q2 — 새 발송 → 이름·템플릿·그룹·즉시 → 미리보기 자동 ✅ PASS

- 미리보기 함수가 채널·템플릿명·그룹명·예상 수신자 수·샘플 5명·렌더링 본문 1건·경고 한 번에 반환
- 비활성 템플릿/그룹 시 경고 추가, 카카오 채널 시 사전 심사 안내, 0명 그룹 시 경고

### Q3 — 등록 → detail 이동, status `pending` → `processing` ✅ PASS

- 등록 시점에는 작업 행 INSERT만, 수신자 INSERT는 cron이 처리(부하 분산)
- cron 1단계가 `scheduledAt` 도래한 pending을 픽업 → 그룹 resolve → 수신자 스냅샷 INSERT → status `processing`

### Q4 — 1~2분 후 폴링 → 진행률·수신자 sent/failed ✅ PASS

- 진행률 폴링 함수가 totalRecipients·success_count·failure_count·status 반환 (가벼운 응답)
- cron 2단계가 chunk 50건씩 어댑터 호출 → 수신자 status 업데이트 → 작업 카운터 누적
- 잔여 0이면 자동 `completed`

### Q5 — 진행 중 작업 취소 → `cancelled` + 미발송 수신자 ✅ PASS

- 취소 함수가 작업 상태 `cancelled` + 미발송 수신자(`pending`/`sending`) 일괄 `cancelled`
- 이미 `sent`/`failed` 처리된 수신자는 보존 (감사 무결성)
- 상태가 `pending`/`processing`이 아니면 거부

### Q6 — 비활성 템플릿 등록 시도 → 거부 ✅ PASS

- 미리보기 응답에 "템플릿이 비활성 상태입니다." 경고
- 등록 시 검증 헬퍼가 비활성 템플릿 거부 (`validation.ok=false`)

### Q7 — 0명 그룹 등록 → 즉시 `completed` ✅ PASS

- 미리보기에 "수신자 그룹에 회원이 0명입니다." 경고
- 등록 자체는 가능(설계 의도) → cron이 그룹 resolve 시 0건 발견하면 `total_recipients=0` + `completed` 즉시 마킹

### Q8 — 예약 시각 과거 입력 → 거부 ✅ PASS

- 검증 헬퍼가 `scheduledAt > NOW()` 강제 (즉시 발송이 아닌 경우)
- 클라이언트에서도 사전 검증

### Q9 — 예약 발송 시각 도래 → cron 픽업 ✅ PASS

- cron 1분 단위 실행 (`* * * * *`)
- 1단계 SELECT: `status='pending' AND (scheduled_at IS NULL OR scheduled_at <= NOW())` → 도래한 작업 자동 픽업
- pending 픽업 한도 10개 (한 번에 너무 많은 작업이 동시에 시작되지 않도록)

### Q10 — 수신자 필터 (성공/실패/대기) ✅ PASS

- 수신자 명단 함수가 status 쿼리 파라미터 받아 필터링
- 페이지네이션도 함께 (limit·offset)

### Q11 — 같은 템플릿+그룹으로 두 작업 등록 → 격리 ✅ PASS

- 작업별 `job_id`로 수신자 스냅샷 분리 INSERT
- cron의 chunk 처리도 `WHERE job_id = X AND status = 'pending'`로 작업 단위 격리
- 두 작업 동시 처리 가능 (processing 동시 한도 5개 안에서)

### Q12 — 검색·기간·상태 동시 필터 ✅ PASS

- 명단 함수가 q + 날짜 범위 + 상태 필터 AND 결합
- 페이지네이션 함께 동작

## 5. cron 동작 검증 (설계서 §4.3)

| 항목 | 점검 결과 |
|---|---|
| 1분 단위 실행 | ✅ schedule `* * * * *` (Netlify Scheduled Function) |
| pending → processing 전환 | ✅ 1단계 startJob에서 scheduledAt 도래·활성 템플릿/그룹·resolve 후 INSERT + status 갱신 |
| chunk 50건 정확 처리 | ✅ CHUNK_SIZE 상수 50, 각 chunk마다 `LIMIT 50` |
| 다음 cron이 이어받음 | ✅ status가 `processing`인 채로 다음 cron tick에 다시 SELECT → 잔여 pending 50건 픽업 |
| 동시 작업 5개 한도 | ✅ PROCESSING_JOB_LIMIT 5, processing SELECT에 `LIMIT 5` |
| pending 픽업 10개 한도 | ✅ PENDING_PICKUP_LIMIT 10 |
| race 방지 | ✅ chunk 처리 시 수신자 행을 `pending → sending` 단일 UPDATE로 마킹, 영향 행 0이면 다른 cron tick이 가져간 것으로 보고 스킵 |
| 잔여 0이면 자동 completed | ✅ chunk 처리 후 잔여 pending/sending 카운트 → 0이면 자동 completed |

→ 1시간 처리 가능량: 5작업 × 50건/min × 60min = **15,000건/시간** (설계서 §4.5와 일치)

→ Swain이 1000명 작업 등록 시 약 **20분** 후 완료 — 설계서 §4.3 명시와 일치

## 9. Swain 직접 클릭 검증이 필요한 항목

세션 쿠키 흐름이 필요. 코드 정합성으로는 통과지만 다음은 클릭으로 최종 확인 권장:

1. 어드민 → 사이드바 [📤 발송 작업] → 빈 목록 또는 기존 작업 표시
2. 새 발송 → 이름·템플릿·그룹·즉시 → 미리보기에 채널·N명·샘플 1건 표시
3. 등록 → 상세 페이지 이동 → 1~2분 후 진행률 증가 + 수신자 sent/failed 표시
4. 진행 중 작업 취소 → 미발송 수신자만 cancelled, sent/failed 보존
5. 비활성 템플릿 등록 → 거부 토스트
6. 0명 그룹 등록 → 즉시 completed
7. 예약 시각 과거 → 거부 토스트
8. 예약 발송 시각 도래 → cron 픽업 → processing 전환
9. 수신자 필터(전체/성공/실패/대기) → 결과 정확
10. 같은 템플릿+그룹으로 두 작업 등록 → 둘 다 정상 발송, 격리
11. 검색·기간·상태 동시 필터 → 결과 정확
12. **진행률 폴링 응답 시간** 0.3초 미만 (실 인증 세션 측정)
13. **cron 실행 시간** 10초 미만 (Netlify 함수 로그 확인)
