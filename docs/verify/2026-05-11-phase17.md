## 2. 발견한 BUG

### BUG-17-01 [HIGH] 비활성 자동 로그아웃이 존재하지 않는 주소 호출 → 서버 쿠키 미삭제

- **증상**: 28분 비활성 → 2분 경고 → 30분 시 자동 로그아웃 흐름에서, 로그아웃 처리 호출이 존재하지 않는 주소로 발송됨. 브라우저는 로그인 페이지로 이동하지만 서버에서 쿠키가 살아 있음. 어드민이 즉시 어드민 화면으로 다시 들어가면 로그인 상태가 유지되어 자동 로그아웃의 보안 의미가 사라짐.
- **원인**: 비활성 감시 스크립트가 어드민 로그아웃 주소를 잘못 적은 채 운영 (실제 등록된 어드민 로그아웃 주소가 아님)
- **수정**: 실제 어드민 로그아웃 주소로 교체
- **파일**: `public/js/admin-idle-guard.js:20`

### BUG-17-02 [MEDIUM] 보안·감사 로그 메뉴 클릭 시 페이지 상단 제목이 "관리자"로 표시

- **증상**: 사이드바에서 "🔒 보안·감사 로그"를 클릭하면 본문은 정상 렌더되지만, 페이지 상단 큰 제목이 "관리자"로 표시됨 (다른 페이지는 메뉴별 제목이 정상)
- **원인**: 어드민 페이지 제목 사전(메뉴 키 → 제목 텍스트 매핑) 안에 보안·감사 로그 항목이 누락
- **수정**: 사전에 `'security-audit': '🔒 보안·감사 로그'` 추가
- **파일**: `public/js/admin.js:36`

### BUG-17-03 [HIGH] 통계 카드 "로그인 실패 횟수"가 항상 0으로 표시

- **증상**: 어드민·사용자 모두 로그인 실패가 발생해도 통계 카드 "로그인 실패 횟수" 값이 0에서 변하지 않음
- **원인**: 통계 집계 시 "로그인" 활동의 success=false만 카운트했으나, 실제로는 로그인 실패가 별도 활동 종류 이름으로 기록됨 (사용자 측 `login_failed`, 어드민 측 `admin_login_failed` 등). 활동 종류명 자체가 다르기 때문에 매칭되지 않음
- **수정**: 실제 기록되는 모든 로그인 실패 종류를 카운트 대상에 포함 (login_failed / admin_login_failed / admin_login_blocked / login_locked / login_blocked + 기존 "login + 실패" 조합)
- **파일**: `netlify/functions/admin-audit-stats.ts:68-82`

## 3. 미완성 영역 (후속 작업 권고)

본 라운드에서 fix하지 않고 후속 작업으로 권고하는 영역. 시나리오 PASS에 영향이 없거나(BUG-17-04·05), 영향 범위가 커서 단독 라운드가 필요함(BUG-17-06).

### BUG-17-04 [LOW] 신규 감사 로그 저장 시 세션 ID·위험 등급 컬럼 미기록

- **증상**: 설계서 §1.1에 따라 감사 로그 테이블에 세션 ID·위험 등급 컬럼이 추가되었지만, 감사 로그를 저장하는 공용 헬퍼에서 해당 컬럼을 채우는 부분이 주석 처리되어 있음. 모든 신규 로그가 두 컬럼을 빈 값으로 저장
- **영향**: 화면상의 위험 등급 표시는 활동 종류 기준 자동 매핑으로 동작 중이라 시나리오 PASS. 단, 향후 DB 기준 필터·통계로 확장 시 깨짐
- **권고**: `lib/audit.ts`의 INSERT 부분에서 세션 ID·위험 등급 필드 활성화. 활동 종류별 자동 등급 산출 로직(이미 작성된 `resolveRiskLevel`)도 같이 켜기

### BUG-17-05 [LOW] 회원 테이블의 연속 로그인 실패 카운트 컬럼이 사용되지 않음

- **증상**: 설계서 §1.1에 따라 회원 테이블에 "연속 로그인 실패 횟수" 컬럼이 추가되었지만, 로그인 실패·성공 로직 어디에서도 이 컬럼을 증가/리셋하지 않음
- **영향**: 기존 누적 실패 카운트(`loginFailCount`)는 동작하므로 잠금 기능에는 문제 없음. 다만 설계 의도였던 "연속 실패 추적"이 동작하지 않음
- **권고**: 어드민/사용자 로그인 함수에서 실패 시 +1, 성공 시 0 리셋 추가

### BUG-17-06 [MEDIUM, 영향 범위 큼] 어드민 회원 화면에 마스킹이 적용되어 있지 않음

- **증상**: 설계서 §3.3·§0의 "어드민 화면 표시 마스킹"이 화면에 반영되지 않음. 백엔드에 마스킹 헬퍼는 만들어져 있으나 회원 목록 API·회원 상세 화면 어디에서도 호출되지 않음
- **영향**: Q5 시나리오 FAIL. 어드민이 회원 전화번호·주민번호를 그대로 보게 됨
- **권고**: 별도 라운드로 어드민 회원 목록·상세 응답에서 전화번호·주민번호를 마스킹 처리. 영향 받는 파일이 회원 목록·상세·검색 등 여러 군데이므로 단독 작업 단위로 분리 권고
