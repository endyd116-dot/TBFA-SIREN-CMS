## 2. 발견한 BUG 6건 (상세)

### BUG-17-01 [HIGH · 해결] 비활성 자동 로그아웃이 존재하지 않는 주소 호출 → 서버 쿠키 미삭제

- **증상**: 28분 비활성 → 2분 경고 → 30분 시 자동 로그아웃 흐름에서, 서버 측 로그아웃 처리 호출이 존재하지 않는 주소로 발송됨. 브라우저는 로그인 페이지로 이동하지만 서버에서 쿠키가 살아 있어, 어드민이 즉시 어드민 화면으로 다시 들어가면 로그인 상태가 유지됨. 자동 로그아웃의 보안 의미 자체가 사라짐
- **원인**: 비활성 감시 화면 스크립트가 어드민 로그아웃 주소를 잘못 적은 채 운영 (실제 등록된 어드민 로그아웃 주소가 아닌 임의 경로)
- **수정**: 실제 등록된 어드민 로그아웃 주소로 교체
- **수정 파일**: `public/js/admin-idle-guard.js`
- **수정 시점**: 1차 라운드 (`verify/phase17` @ eccc45d)

### BUG-17-02 [MEDIUM · 해결] 보안·감사 로그 메뉴 클릭 시 페이지 상단 제목이 "관리자"로 표시

- **증상**: 사이드바에서 "🔒 보안·감사 로그"를 클릭하면 본문은 정상 렌더되지만, 페이지 상단 큰 제목이 메뉴명 대신 "관리자"로 표시됨 (다른 페이지는 메뉴별 제목이 정상)
- **원인**: 어드민 페이지 제목 사전(메뉴 키 → 제목 텍스트 매핑) 안에 보안·감사 로그 항목이 누락
- **수정**: 사전에 보안·감사 로그 항목 추가
- **수정 파일**: `public/js/admin.js`
- **수정 시점**: 1차 라운드 (`verify/phase17` @ eccc45d)

### BUG-17-03 [HIGH · 해결] 통계 카드 "로그인 실패 횟수"가 항상 0으로 표시

- **증상**: 어드민·사용자 모두 로그인 실패가 실제로 발생해도 통계 카드의 "로그인 실패 횟수" 값이 0에서 변하지 않음. 실제 운영 보안 모니터링 지표가 무력화
- **원인**: 통계 집계 시 "로그인" 활동의 실패만 카운트했으나, 실제로는 로그인 실패가 별도 활동 종류 이름으로 기록됨 (사용자 측은 `login_failed`, 어드민 측은 `admin_login_failed` 등). 활동 종류명이 다르기 때문에 매칭되지 않음
- **수정**: 실제 기록되는 모든 로그인 실패 종류를 카운트 대상에 포함 (사용자 실패 / 어드민 실패 / 어드민 차단 / 잠금 / 정지·탈퇴 차단 + 기존 "login + 실패" 조합)
- **수정 파일**: `netlify/functions/admin-audit-stats.ts`
- **수정 시점**: 1차 라운드 (`verify/phase17` @ eccc45d)

### BUG-17-04 [LOW · 해결] 신규 감사 로그 저장 시 세션ID·위험등급 컬럼 미기록

- **증상**: 설계서 §1.1에 따라 감사 로그 테이블에 세션 ID·위험 등급 컬럼이 추가되었지만, 감사 로그를 저장하는 공용 헬퍼에서 해당 컬럼을 채우는 부분이 주석 처리. 모든 신규 로그가 두 컬럼을 빈 값으로 저장
- **영향**: 화면상의 위험 등급 표시는 활동 종류 기준 자동 매핑으로 동작 중이라 1차 시나리오 PASS. 단, 향후 DB 기준 필터·통계로 확장 시 깨짐
- **수정**: 감사 로그 저장 헬퍼에서 두 컬럼 INSERT 활성화
  - 세션ID: 요청 쿠키의 토큰을 SHA-256으로 해시한 앞 32자 사용 (토큰 자체는 저장하지 않음, 같은 로그인 세션이면 같은 식별자)
  - 위험등급: 활동 종류별 사전(중요 작업은 critical/high, 단순 조회는 low)을 통한 자동 매핑
- **수정 파일**: `lib/audit.ts`
- **수정 시점**: 후속 라운드 (`fix/phase17-followup` @ 66f2b93)

### BUG-17-05 [LOW · 해결] 회원 테이블의 연속 로그인 실패 카운트 컬럼이 사용되지 않음

- **증상**: 설계서 §1.1에 따라 회원 테이블에 "연속 로그인 실패 횟수" 컬럼이 추가되었지만, 로그인 실패·성공 로직 어디에서도 이 컬럼을 증가/리셋하지 않음
- **영향**: 기존 누적 실패 카운트는 동작하므로 잠금 기능 자체에는 문제 없음. 다만 설계 의도였던 "지금 시점에서 연속 실패 N회"를 정확히 보여주는 지표가 사용 불가
- **수정**: 어드민·사용자 로그인 함수 양쪽에서 — 비밀번호 불일치 시 +1, 로그인 성공 시 0 리셋
- **수정 파일**: `netlify/functions/admin-login.ts`, `netlify/functions/auth-login.ts`
- **수정 시점**: 후속 라운드 (`fix/phase17-followup` @ 66f2b93)

### BUG-17-06 [MEDIUM · 🚚 Phase 21 이관] 어드민 회원 화면에 마스킹이 적용되어 있지 않음

- **증상**: 설계서 §3.3·§0의 "어드민 화면 표시 마스킹"이 화면에 반영되지 않음. 백엔드에 마스킹 헬퍼는 만들어져 있으나 회원 목록·상세·익스포트·후원자 관리·효성 관리 등 어디에서도 호출되지 않음
- **Q5 시나리오 결과**: FAIL — 어드민이 회원 전화번호를 그대로 보게 됨 (단, 회원 테이블에 주민번호 컬럼은 존재하지 않으므로 마스킹 대상은 사실상 전화번호 단독)
- **원인**: 백엔드는 응답에 원본 전화번호를 그대로 포함, 프론트도 가공 없이 표시
- **본 라운드 조치**: 코드 수정 없이 영향 파일 조사만 완료. 정식 설계 후 단독 라운드(Phase 21)로 처리 권고. 이유는 §3 영향 범위 참고
- **수정 시점**: 본 라운드 미수정 — Phase 21로 이관

## 6. 후속 권고 사항

### 6.1 Phase 21 — 어드민 화면 마스킹 정식 설계 (BUG-17-06 이관)

본 보고서 §3 조사 결과를 기반으로 별도 라운드 진행 권고. 다음 항목 결정 필요:

1. **권한별 분기 정책 결정**
   - super_admin: 원본 그대로 표시
   - 일반 admin / 운영자: 마스킹 표시
   - 또는: 모두 마스킹하되 클릭·확장 시 원본 노출(감사 로그 기록 동반)

2. **그룹별 적용 정책 결정** (§3.1 5개 그룹)
   - 효성 CMS+ 그룹은 **마스킹 절대 불가** — 무조건 원본 유지
   - 익스포트 그룹은 권한 제한(super_admin 전용)이 더 적절할 수 있음
   - 익명 신고 추적(`admin-anonymous-reveal`)은 별도 감사 기록과 결합한 정책 필요

3. **백엔드 vs 프론트 마스킹 위치 결정**
   - 백엔드 마스킹: 응답 본문 자체가 안전, 단 익스포트와 충돌
   - 프론트 마스킹: 응답에 원본이 있어 개발자 도구로 우회 가능, 단 익스포트는 영향 없음

4. **헬퍼 통합 정책 결정**
   - 백엔드 헬퍼(`lib/masking.ts`)와 프론트 인라인 마스킹이 분리되어 있는 현재 구조의 통합 여부

### 6.2 (선택) 보안·감사 로그 화면 가독성 개선

- 활동 종류 라벨 사전에 `login_failed`·`admin_login_failed`·`admin_login_blocked` 등 실제 활동 종류명을 추가해 화면에 한글 라벨로 표시 (현재는 영문 그대로 노출)

### 6.3 (선택) 감사 로그 위험 등급 DB 기준 필터

- BUG-17-04 fix로 신규 로그는 위험 등급 컬럼에 값이 채워짐. 향후 화면 필터를 활동 종류 기반 매핑이 아닌 DB 컬럼 직접 조회로 전환하면 더 정확한 통계 가능 (기존 로그는 NULL이므로 일정 기간 후 전환)
