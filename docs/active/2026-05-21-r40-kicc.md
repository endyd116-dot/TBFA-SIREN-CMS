## §0 요구사항 확정 (Swain 결정)

| 항목 | 결정 | 근거 |
|---|---|---|
| 전환 방식 | **전면 즉시 전환** (옵션 B) | 오픈 전·실제 토스 후원자 0명(현 데이터 전부 테스트) → 깔끔한 컷오버 |

| DB 전략 | **기존 테이블 재활용 + `toss*`→범용(`pg*`) 컬럼 재정의** | 테스트 데이터 폐기 가능. 두 체계 병존·이중관리 회피. 향후 PG 교체도 쉬움 |

| 토스 회원 마이그 | **불필요** | 실데이터 없음. 토스 빌링키→KICC 변환 로직 안 만듦 |
| 모델 분배 | **A·B·C 전부 Opus 4.7** | 결제 라운드 — 미묘한 버그가 돈에 직결. 표준(B=Sonnet) 대비 상향 |

**핵심 원칙**: 컬럼·함수·엔드포인트는 **PG 비종속(provider-agnostic)** 네이밍으로 정리한다. `tossPaymentKey`처럼 PG명이 박힌 이름 대신 `pgTid`·`pgProvider` 등으로 → 다음 PG 교체 시 또 전면 개명하지 않도록.

## §1 KICC 명세 핵심 (kicc.md 정독 요약)

- **API 도메인**: 개발 `https://testpgapi.easypay.co.kr` / 운영 `https://pgapi.easypay.co.kr` (TLS 1.2, 30초 타임아웃)
- **인증**: `mallId`(8byte 가맹점ID) + `msgAuthValue`(HmacSHA256). 토스의 Basic 인증과 완전히 다름.
- **일시 결제** (리다이렉트 기반):
  1. 거래등록 `POST /api/ep9/trades/webpay` → 응답 `authPageUrl`
  2. 결제창 호출(브라우저 리다이렉트) → 사용자 결제 → `returnUrl`로 복귀(`authorizationId` 수신)
  3. 승인 `POST /api/ep9/trades/approval` (mallId·shopTransactionId·authorizationId·shopOrderNo·approvalReqDate) → 응답 `pgCno`(거래고유번호·취소키)·`amount`(검증필수)·`statusCode`·`paymentInfo.cardInfo`
- **정기 결제(빌키)**:
  1. 빌키 등록창 `POST /api/ep9/trades/webpay?clientTypeCode=81` → `authPageUrl`
  2. 등록창 호출 → `returnUrl`(`authorizationId`)
  3. 빌키 발급 `POST /api/ep9/trades/approval` → 응답 **`cardNo`(빌키, 60자)**
  4. 빌키 자동결제 `POST /api/trades/approval/batch` (`billKeyMethodInfo.batchKey`·`cardMethodInfo.installmentMonth`·amount·approvalReqDate)
  5. 빌키 삭제 `POST /api/trades/removeBatchKey` (batchKey·removeReqDate)
- **취소/환불**: `POST /api/trades/revise` (pgCno·reviseTypeCode·부분취소 amount). 환불은 실시간/지결(D+1·웹훅) 2종.
- **조회**: `POST /api/trades/retrieveTransaction` (shopTransactionId·transactionDate, 전일~당일만)
- **웹훅(노티)**: HTTP POST·10 이벤트(승인10·변경20·입금30·환불완료50 등)·3분 간격 최대10회 재전송·**서명검증 없음**(resCd `0000` ack, 실패 `5001`)·**shopOrderNo/pgCno 기반 중복방지 필수**
- **갭(가맹점 자체구현)**: 빌키 조회/유효성검증 API 없음·자동재시도 없음 → SIREN 기존 `billingLogs`·`nextChargeAt`·`consecutiveFailCount`·재시도 스케줄러가 그대로 메꿈(구조 적합).

## §2 DB 설계 (B 담당 — 기존 테이블 재활용·컬럼 재정의)

> ⚠️ 테스트 데이터 폐기 전제. B는 schema.ts 정독 후 정확한 컬럼 확정 → 마이그 함수 작성 → Swain 호출 → schema 활성화 순서 준수(§9.1·§9.2). schema.ts는 append-only 헤더 명시.

### 컬럼 재정의 매핑

**`billingKeys`** (정기 결제 빌키):
| 현재(toss) | 변경(범용) | 비고 |
|---|---|---|
| `billingKey` | `billingKey` (유지·의미 변경) | KICC 빌키(cardNo) 저장. 컬럼명 churn 회피 |
| `customerKey` | **제거 또는 nullable 보존** | KICC엔 customerKey 개념 없음. 내부 회원 식별은 memberId로 충분 |
| `cardCompany`·`cardNumberMasked`·`cardType` | 유지 | KICC `paymentInfo.cardInfo`에서 채움 |
| `nextChargeAt`·`billingDay`·`consecutiveFailCount`·`cardExpiryMonth`·`isActive`·`reactivatedAt`·`deactivatedAt` | 유지 | PG 비종속 |
| (신규) | `pgProvider` default `'kicc'` | 향후 다중 PG 대비 |

**`donations`**:
| 현재 | 변경 | 비고 |
|---|---|---|
| `tossPaymentKey` | `pgTid` | KICC pgCno 저장 (취소키) |
| `tossOrderId` | `pgOrderNo` | KICC shopOrderNo |
| `billingKeyId`·`billingLogId`·`type`·`status` | 유지 | |
| (신규) | `pgProvider` | 'kicc'/'hyosung'/'bank' 구분 |

**`billingLogs`**:
| 현재 | 변경 |
|---|---|
| `tossOrderId` | `pgOrderNo` |
| `tossPaymentKey` | `pgTid` |
| `tossResponseCode` | `pgResponseCode` |
| `tossResponseMessage` | `pgResponseMessage` |
| 그 외(memberId·attemptType·attemptNumber·amount·status·errorDetail·donationId·nextRetryAt) | 유지 |

**`cardExpiryAlerts`**: 컬럼 변경 없음(빌키·만료월 PG 비종속).

**enum**: `donationStatusEnum`·`donationTypeEnum` 변경 없음.

### 마이그레이션
- `migrate-r40-kicc-columns.ts` — 컬럼 rename(`ALTER ... RENAME COLUMN`)·신규 컬럼 add(`pgProvider`)·테스트 결제데이터 정리(`DELETE FROM billing_logs/donations WHERE ...` 또는 truncate). 멱등 보장.
- 호출 후 즉시 삭제(§9.2).

## §3 백엔드 설계 (B 담당)

### 신규 공용 라이브러리 `lib/kicc.ts` (구 `lib/toss-billing.ts` 대체)
PG 비종속 헬퍼는 그대로 이식, KICC 전용은 신규:
- `getKiccConfig()` — mallId·secret·domain (test/live)
- `signMsgAuth(parts)` — HmacSHA256 msgAuthValue 생성·검증
- `registerTrade(params)` — `POST /api/ep9/trades/webpay` (일시·빌키등록 공용, clientTypeCode 분기) → authPageUrl
- `approveTrade(params)` — `POST /api/ep9/trades/approval` (일시승인·빌키발급 공용) → pgCno 또는 cardNo
- `chargeWithBillingKey(params)` — `POST /api/trades/approval/batch` (정기 자동결제) → ChargeResult
- `cancelPayment(pgCno, reviseTypeCode, amount?)` — `POST /api/trades/revise`
- `removeBillingKey(batchKey)` — `POST /api/trades/removeBatchKey`
- `retrieveTransaction(shopTransactionId, date)` — 조회
- `normalizeKiccError(resCd)` — KICC resCd → ErrorCategory(card_invalid/insufficient_funds/declined/network/rate_limit/unknown)·retryable 매핑
- **이식(불변)**: `generateBillingOrderId`(→shopOrderNo 40자)·`calculateNextRetryAt`·`calculateNextBillingDate`·`getCurrentYearMonth`·`logBillingAttempt`·`logBillingResult`(키명 pg*로)

## §4 프론트 설계 (A 담당)

### 결제 방식 전환: SDK 팝업 → 리다이렉트 결제창
토스는 `js.tosspayments.com` SDK 팝업이었으나 **KICC는 authPageUrl 리다이렉트** 방식.

| 파일 | 변경 |
|---|---|
| `public/js/donate.js` | 토스 SDK 로드·`requestPayment` 제거 → 백엔드 `donate-kicc-register` 호출로 authPageUrl 받아 `location.href` 이동(또는 form POST). 일시/정기 분기 유지 |
| `public/billing-register.html`·`.js` | 빌키 등록 → `billing-register`(거래등록) → authPageUrl 리다이렉트 |
| `public/billing-success.html`·`billing-fail.html` | KICC returnUrl 콜백 수신·결과 표시 (authorizationId 파라미터 처리) |
| `public/payment-success.html`·`payment-fail.html` | 일시결제 returnUrl 콜백 |
| `public/campaign.html`·`campaigns.html` | 후원 폼이 donate.js 통해 동작 → donate.js 변경에 따라감 |

캐시버스터: 변경 JS·HTML 참조 `?v=` 일괄 갱신.

### returnUrl 흐름 (중요·A·B 공유 계약)
1. 결제창에서 결제 완료 → KICC가 `returnUrl`로 POST/GET (authorizationId·shopOrderNo)
2. returnUrl 페이지(success/fail.html)가 authorizationId를 백엔드 approve 함수로 전달
3. 백엔드 승인 API 호출 → 성공/실패 → 화면 표시

## §6 작업 분배·트리거 (메인이 A·B 완료 보고 후 머지)

**의존 순서**: B(스키마·lib·API·웹훅) → A(프론트는 B의 returnUrl·응답 계약에 맞춤, B 머지 전엔 §6.2 계약 mock으로 진행) → C(둘 다 머지 후 KICC 테스트 키로 검증).

**머지 순서 (GUIDE §5 강제 — 마이그 게이팅)**:
```
1. B 완료 보고 → 메인 키 대조 + git log origin/main..feature/r40-kicc-back 확인 → main 머지
2. (마이그 있음) Swain께 /api/migrate-r40-kicc-columns?run=1 안내 → success 확인
3. 메인이 schema 정의 활성화 + 마이그 파일 삭제
4. A 완료 보고 → 영역 충돌 0 확인(git diff main...front -- lib db netlify = empty) → main 머지
5. B+A 묶어 1회 push (§9.3 배치) → 배포
6. C 검증 트리거 → 검증·fix → 머지 → 마감
```
✅ 배포 경로(Swain 확정): **main 직결 + KICC_MODE=test** (테스트 키라 실결제 0·main 배포에서 검증 후 live 전환). GUIDE §11 dev→main 2단계는 R40에선 휴면.
✅ 모델(Swain 확정): **A·B·C 전부 Opus 4.7** (결제 라운드 — 표준 B=Sonnet보다 상향).

> 트리거 전문은 본 설계 확정(스키마 컬럼명·응답 키 최종 합의) 후 §6.1(B)·§6.2(A)·§6.3(C)에 체크박스 양식으로 작성. **현재는 B가 schema.ts·기존 함수 본문 정독 후 컬럼명·응답 키를 메인과 1차 합의하는 단계가 선행**(§9.1.9 — 일반 가정 금지).
