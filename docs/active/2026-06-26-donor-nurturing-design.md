# 후원자 단계별 자동 너처링 시스템 — 최종 설계 (v1.0 draft)

> 2026-06-26 · Swain 요구 + 외부 모범사례 검색 + 내부 시스템 분석 종합.
> 목표: 정기·예비·잠재 후원자를 **각 탭**으로 두고, **D0~D365 타임라인**으로 신뢰·브랜드 가치를 쌓고,
> **D365 이후에도 중요한 알림은 영구히** 자동 발송. 단일 기능·운영자가 어드민에서 직접 관리(§6.18).

---

## 1. 컨셉

**"후원자 여정 자동화" 단일 기능.** 싸이렌 어드민에 새 영역. 후원자 단계(세그먼트)별로 가입/분류 시점(D0)부터
1년(D365)까지 무엇을·언제·어떤 채널로 보낼지 타임라인으로 설계하고, 그 이후는 영구 반복 규칙으로 유지.

- **탭 3개**: ① 정기 후원자 ② 예비 후원자(일시/이탈 2 트랙) ③ 잠재 후원자
- **타임라인**: 각 탭마다 D0~D365 단계(step) 자유 배치
- **Evergreen(영구)**: D365 이후 지속 규칙(분기 리포트·기념일·연말 등) — 끝나지 않음
- **전환 시 자동 이동**: 예비→정기 전환되면 예비 여정 종료 + 정기 여정 진입(분류 동기화 연동). 잠재→첫 후원 시 예비로.
- **재사용**: 실제 발송은 기존 발송 엔진(communication_send_jobs/recipients + 디스패처)·템플릿·동의 체크 그대로. 신규는 "오케스트레이션 층"만.

## 3. 데이터 모델 (신규 5 테이블)

| 테이블 | 역할 |
|---|---|
| `nurture_journeys` | 세그먼트별 여정(=탭) 정의. segment(regular/prospect_onetime/prospect_cancelled/potential)·is_active·진입기준 |
| `nurture_steps` | 여정의 단계. journey_id·day_offset(0~365)·channel·template_id·conditions(jsonb)·is_active·label·sort |
| `nurture_evergreen_rules` | D365+ 영구 규칙. journey_id·cadence(monthly/quarterly/anniversary/yearend)·channel·template_id·conditions·is_active |
| `nurture_enrollments` | 회원×여정 진행상태. member_id·journey_id·enrolled_at(=D0)·status(active/converted/exited/completed)·converted_at |
| `nurture_sends` | 단계별 발송 로그(멱등·중복방지·KPI). enrollment_id·step_id(or evergreen_rule_id)·sent_at·job_id·status |

- 발송 자체는 기존 `communication_send_jobs`/`recipients` 재사용(job_id로 연결).
- 멱등 핵심: (enrollment_id, step_id) UNIQUE → 같은 단계 두 번 안 보냄.

## 4. 엔진 (cron-nurture-runner · 매일 1회 · :00 정렬)

매일:
1. **진입(enroll)**: 각 세그먼트 현재 구성원 중 enrollment 없는 회원 → enrolled_at=분류일(donor_evaluated_at)로 enrollment 생성.
2. **단계 발송**: 각 활성 step(day_offset=N)에 대해 `enrolled_at + N일`이 오늘인(±1일 윈도우) enrollment 중 ① 미발송(nurture_sends 없음) ② 동의 OK ③ status=active ④ 조건 충족 → 발송 큐 적재(기존 디스패처가 즉시 처리).
3. **Evergreen**: D365 경과(또는 타임라인 완료) enrollment → cadence 규칙 충족 시 발송(분기/기념일/연말).
4. **전환·종료**: 회원 세그먼트가 바뀌면(예비→정기 등) 기존 enrollment status=converted/exited + 새 세그먼트 여정 enroll. (분류 동기화 cron과 연동)
5. **안전**: 수신 동의·블랙 차단 게이트(기존 로직 재사용), **하루 1통·주 N통 빈도 상한**, quiet hours, 비용 보호(기본 채널 메일·문자/알림톡은 핵심 단계만).

> 진입 기준일(D0): 예비/정기/이탈은 "그 세그먼트로 분류된 날", 잠재는 "리드 등록일". 전환되면 새 여정 D0 재시작.

## 5. 세그먼트별 기본 여정 (시드 — 외부 모범사례 매핑)

### ① 잠재 후원자 → 첫 후원 (요청은 뒤로·신뢰 먼저)
| 시점 | 채널 | 의도 | 요청 |
|---|---|---|---|
| D0 | 메일 | 환영 + 미션(왜 존재하는가) | ❌ |
| D+4 | 메일 | 유족 지원 실제 사례 | ❌ |
| D+10 | 메일 | 왜 지금 중요한가(맥락·공감) | ❌ |
| D+18 | 메일/알림톡 | 돈 아닌 참여(행사·서명·SNS) | △ |
| D+26 | 메일+문자 | **첫 후원 권유**(구체 금액) | ✅ |
| Evergreen | 월간 | 뉴스레터 + 연말/기념일 소프트 어스크 | △ |
*첫 후원 시 → 예비-일시 여정으로 자동 이동.*

### ② 예비-일시(1회 후원) → 정기 전환 ★최고 ROI
| 시점 | 채널 | 의도 |
|---|---|---|
| D0 | 메일 | 감사 + 영수증 |
| D+2 | 메일 | "당신의 1만원이 한 일" 임팩트 |
| D+7 | 알림톡 | 유족 후기 |
| **D+14** | 메일+문자 | **정기 전환 권유**(원클릭) |
| D+30 | 알림톡 | 미전환 시 2차 넛지 |
| Evergreen | 분기 | 소식·재후원 유도 |

### ②' 예비-이탈(전 정기·해지) → 재활성화
| 시점 | 채널 | 의도 |
|---|---|---|
| D0 | 메일 | 감사 + 사유 청취(1클릭 설문) |
| 조건: 카드만료/결제실패 | 문자+알림톡 | **결제정보 갱신 링크**(자동 감지) |
| D+30 | 메일 | 그간 임팩트 + 재개 권유 |
| D+60/90 | 메일/문자 | 마지막 윈백(기념일·캠페인) |
| Evergreen | 분기 | 관계 유지 |

### ③ 정기 후원자 → 유지·상향
| 시점 | 채널 | 의도 |
|---|---|---|
| D0(첫 정기) | 메일+알림톡 | 환영·약속 |
| 분기 | 메일 | 임팩트 리포트(**non-ask**) |
| 기념일(6개월/1년) | 메일+알림톡 | 감사 + 연간 영수증 + (선택)상향 제안 |
| 조건: 이탈점수 임계 | 자동 | **자동 재유치**(현 수동→자동) |
| 조건: 카드만료 임박 | 알림톡 | 결제정보 갱신 |
| Evergreen | 분기·기념일 | 영구 스튜어드십 |

## 6. 기존 시스템 연결·흡수

- **흡수**: 기존 이벤트 트리거(donation_complete welcome 등) 중 여정과 겹치는 건 여정 D0로 통합(중복발송 방지). 생일·기념일 트리거는 여정 조건스텝으로 편입 가능.
- **연동**: churn-predictor 점수 → 정기 여정 "이탈위험" 조건스텝. billing-card-expiry → 결제갱신 스텝. 분류 동기화 cron → 진입/전환.
- **재사용**: 발송엔진·템플릿·동의체크·알림톡 CMS 그대로.

## 7. 안전장치·비용

- 수신 동의(메일/문자/알림톡 각각)·블랙 차단 게이트. 미동의 발송 0.
- 빈도 상한(하루 1·주 N), quiet hours, KST.
- 비용: 기본 메일(ROI 최고), 문자/알림톡은 핵심 전환 단계만. 대량 발송은 디스패처 한도로 분산.
- 미리보기·테스트 발송·발송 전 대상 수 표시.

## 8. KPI 대시보드

세그먼트별 여정 진행률 · 단계별 오픈/클릭 · **전환율(잠재→첫후원, 예비→정기)** · 이탈 회복률 · 정기 retention · 발송량/비용.

## 9. 빌드 단계

- **Phase 1 (MVP·최고 ROI)**: 데이터모델 + 엔진 cron + 어드민 타임라인 편집 + **예비-일시→정기 전환 여정** 시드. (방금 만든 자동 예비등록과 직결)
- **Phase 2**: 정기 스튜어드십 + 이탈 재유치(자동) + 카드갱신 + 잠재 획득 여정.
- **Phase 3**: Evergreen 전 세그먼트 + KPI 대시보드 + (선택)A/B·AI 메시지.
- 각 Phase: 마이그레이션 → 엔진 → 어드민 화면 → 시드 → 검증.
