## 5. 추천: 시나리오 B

### 이유
1. **데드라인-가용 시간 정합성**: 1주 + 30h vs B의 8~12h ≈ 2배 안전 마진 → 회귀·재검증 여유
2. **마일스톤 일관성**: C·D는 같은 도메인(donor_type)을 정착시키는 흐름. 한 사이클에 묶는 게 운영 일관성에 유리. 분리하면 중간 상태가 길어져 도메인 혼란
3. **병렬화 의사 정합성**: 적극 활용 의사 ↔ Phase 2·3에서 A·B 보조 채팅 사용 (full 병렬은 아니지만 보조 활용)
4. **#8 결정**: 15~18h라 한 사이클 안에 의미 있게 시작·종료 어려움. 깨끗이 다음 사이클로 미루는 게 효율적
5. **TS 149건**: 운영 영향 0이라 시급도 낮음. 다음 사이클 자투리

### 시나리오 B를 안 골라야 하는 신호
- 외부 데드라인이 #16 단계 B만으로 충족된다 → A
- 다음 사이클 일정이 매우 빡빡할 예정이라 #8 미리 시작 필요 → C
- schema 회귀 사고 재발 우려 매우 큼 → A
