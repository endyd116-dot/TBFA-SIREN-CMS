## 💬 7. 채팅 시스템 (★ Phase H+I 강화)

### 아키텍처

\```
사용자 마이페이지 → "1:1 상담" 탭 → 카테고리 선택 → 채팅방 생성
    ↓
텍스트/이미지 메시지 송수신 (5초 폴링)
    ↓ (★ H-1: 이미지가 실제로 표시됨, 라이트박스+다운로드)
관리자 백오피스 → "채팅 관리" 메뉴 → 좌우 분할 레이아웃
    ↓
회원 정보 + 메모 + 답변 + 종료/보관/복구/블랙
\```

### 주요 사양

* **폴링**: 5초 간격
* **카테고리**: 후원/홈페이지/가입/기타
* **메시지 보존**: 영구
* **이미지 보존**: 1년 (★ I-2: 자동정리, active방은 보존)
* **이미지 압축**: 클라이언트 Canvas API (max 1200px, JPEG 75%)
* **이미지 표시**: ★ H-1 — 인라인 `<img>` + 라이트박스 + 다운로드
* **상태 전환**: active ↔ closed ↔ archived (★ I-3 채팅 아카이브)
* **블랙리스트**: 관리자 등록/해제 + ★ I-3 회원 모달에 표시
* **채팅 메모**: 관리자 메모 → ★ I-3 회원 모달 목록 표시
* **알림**: 헤더 뱃지 (30초 폴링) + 사이드바 뱃지

## 📄 8. PDF 영수증 시스템 (★ Phase H-2 완전체)

### 핵심 흐름

\```
1. 회원이 후원 → donate.ts → DB 저장 + 감사 메일 (★ H-3)
2. 마이페이지 → 후원 내역 → "📄 발급" 버튼
   ↓
3. /api/donation-receipt?id=N (본인 또는 관리자)
   ↓
4. issueReceiptNumber() — 영수증번호 발급 (TBFA-2026-000042)
   ↓
5. generateReceiptPDF() — DB(receipt_settings) 우선 → 환경변수 → 샘플 폴백
   ↓
6. pdf-lib + NotoSansKR 폰트 임베딩 (subset: false, 한글 안정)
   ↓
7. A4 PDF 바이너리 응답 (인라인 또는 다운로드)
\```

### 영수증 설정 관리 (★ H-2d 양방향 동기화)

\```
[admin.html "📄 영수증 설정" 메뉴]  ←→  [cms-tbfa.html "📄 영수증 설정"]
              ↓                              ↓
          GET/PATCH /api/admin/receipt-settings (단일 진실 원천)
                          ↓
                  receipt_settings 테이블 (id=1)
                          ↓
                  PDF 생성 시 자동 반영
\```

### 한글 깨짐 핫픽스 (중요)

\```ts
// lib/pdf-receipt.ts
let _fontCache: ArrayBuffer | null = null;

function loadKoreanFont(): Uint8Array {
  if (!_fontCache) {
    const buf = readFileSync(fontPath);
    _fontCache = buf.buffer.slice(buf.byteOffset, buf.byteOffset + buf.byteLength);
  }
  return new Uint8Array(_fontCache.slice(0));  // 매번 새 Uint8Array
}

const font = await pdfDoc.embedFont(fontBytes, { subset: false });  // ← 핵심
\```

* `subset: true`는 큰 한글 폰트(6MB)에서 CMAP 누락 → 한글 깨짐
* 폰트 cache는 ArrayBuffer로 보관 + 매번 새 Uint8Array (pdf-lib mutate 방어)
* PDF 크기 ~6MB로 증가하지만 한글 100% 안정

## 💳 6. 결제 시스템 — 듀얼 옵션 모델

### 아키텍처

```
[후원하기 모달]
   ↓
[정기/일시 선택]
   ├── 일시 후원 → 토스 카드 결제 (즉시)
   └── 정기 후원 → 사용자 선택
       ├── 💳 토스 카드 → 빌링키 등록 + 매월 자동 결제
       └── 🏦 효성 CMS+ → 신청 접수 + 관리자 수동 등록
```

### 토스페이먼츠 흐름

**일시 결제:**

```
donate.js → POST /api/donate-toss-prepare (orderId 발급 + DB pending)
         → 토스 SDK requestPayment() (결제창)
         → 사용자 결제 완료
         → /payment-success.html?paymentKey=...&orderId=...&amount=...
         → POST /api/donate-toss-confirm (토스 API confirm + DB completed + 영수증 + 메일)
```

**정기 결제 (빌링키):**

```
donate.js → sessionStorage에 intent 저장 → /billing-register.html
         → POST /api/billing-prepare (customerKey 발급)
         → 토스 SDK requestBillingAuth() (카드 등록 창)
         → /billing-success.html?authKey=...&customerKey=...
         → POST /api/billing-confirm:
           1. 토스 API /billing/authorizations/issue (billingKey 발급)
           2. 토스 API /billing/{billingKey} (첫 결제)
           3. DB: billing_keys INSERT + donations INSERT
           4. 영수증 발급 + 감사 메일
```

**매월 자동 결제 (cron):**

```
cron-billing-monthly.ts (매일 KST 03:00 = UTC 18:00)
  → billing_keys WHERE isActive=true AND nextChargeAt <= 오늘
  → 각 빌링키별 토스 API /billing/{billingKey} 호출
  → 성공: donations INSERT + nextChargeAt 1개월 후
  → 실패: consecutiveFailCount++ (1회:3일후 재시도 / 2회:5일후 / 3회:자동해지)
  → 성공/실패 메일 자동 발송
```

**Webhook (비동기 보장):**

```
POST /api/toss-webhook (토스가 비동기로 호출)
  → HMAC-SHA256 서명 검증 (timingSafeEqual)
  → PAYMENT_STATUS_CHANGED 이벤트 처리
  → DB 자동 동기화 (멱등성 보장)
  → 항상 200 응답 (재시도 방지)
```

### 토스 환경변수 전환

```
테스트 → 운영:
  TOSS_MODE = live              (test → live)
  TOSS_LIVE_CLIENT_KEY = live_ck_...  (추가)

### 효성 CMS+ 연동

**역할 분담 모델:**

```
효성 CMS+ = 결제/청구/수납의 마스터 (계좌이체 자동출금)

## 🔒 7. 보안 설계

### 인증 토큰/쿠키

| 구분               | 토큰      | 쿠키 이름               | 만료                  |
| ---------------- | ------- | ------------------- | ------------------- |
| 사용자 (remember X) | JWT 1d  | `siren_token`       | 세션 쿠키 (maxAge:null) |
| 사용자 (remember O) | JWT 14d | `siren_token`       | 14일 영속              |
| 관리자              | JWT 2h  | `siren_admin_token` | 세션 쿠키               |
| 가입 직후            | JWT 1d  | `siren_token`       | 세션 쿠키               |

### 보안 레이어

* **비밀번호**: bcrypt 10라운드
* **비번 재설정**: SHA-256 해시 토큰, 30분 만료, 1회용, 1시간 3회
* **이메일 인증**: SHA-256 해시 토큰, 24시간 만료, 1시간 5회
* **로그인 잠금**: 5회 실패 → 30분 잠금 (loginFailCount + lockedUntil)
* **타이밍 공격 방어**: `DUMMY_BCRYPT_HASH` (lib/auth.ts)로 응답 시간 균일화
* **Enumeration 방지**: 존재/비존재 이메일에 동일 응답 (비번 재설정)
* **토큰 응답 본문 제거**: A-2 적용 (httpOnly 쿠키만, JSON에 token 없음)
* **토스 webhook 서명**: HMAC-SHA256 + crypto.timingSafeEqual
* **CSRF**: SameSite=Lax 쿠키
* **XSS**: escapeHtml 전수 적용
* **401 자동 처리**: common.js fetch monkey-patching (3초 쿨다운)
* **감사 로그**: 모든 중요 액션 audit\_logs 자동 기록

### 세션 마커 패치 (적용 여부 확인 필요)

동작: sessionStorage에 마커 저장 → 브라우저 종료 시 삭제 → 재시작 시 감지 → 강제 로그아웃
파일: common.js (SIREN_SESSION 모듈), auth.js (markUserLogin), admin.js (markAdminLogin)
